## [v0.9.0] ‚Äî 2026-07-10 ‚Äî ops hardening + cold-start reliability (the machine runs itself)

**The daily tool goes fully autonomous + observable.** Deployed 2026-07-09; **live-proven 2026-07-10 06:00 UTC** ‚Äî the first fully unattended cron ran the whole pipeline end-to-end and delivered a digest with nobody watching (post-deploy smoke gate `200 @ 0006_subscores`, both alarms armed). Agentic squad program **Run 5 + Micro-run 6**.

### Added ‚Äî ops hardening (Squad Run 5)

- **[ADR-0029] S3 remote state ‚Äî the stack's memory stops living on one laptop.** Before: the Terraform state existed only as a local file ‚Äî a lost/reimaged laptop meant hand-cleaning a live **14-resource stack**, and the state holds the **account id, emails, and every ARN**. Now `providers.tf` carries a **`backend "s3"`** block ‚Äî bucket **`jobfetcher-tfstate-198592435375`, deliberately UNMANAGED** (created once outside the stack: the state must **survive `terraform destroy`**; a stack-managed state bucket is a chicken-and-egg ‚Äî destroying the stack would delete the record of its own teardown) ‚Äî with **`use_lockfile = true`** (TF ‚â• 1.10 **native S3 locking**, no DynamoDB lock table; `required_version` bumped 1.6 ‚Üí 1.10 accordingly). The **ONE-TIME human-present migration** (backup-first ‚Üí `terraform init -migrate-state` ‚Üí a **zero-drift `terraform plan` stop condition** ‚Üí local-file removal last) is [deploy runbook ¬ß3](docs/runbooks/deploy.md).
- **Two CloudWatch alarms ‚Üí one SNS topic ‚Üí email** (`var.recipient_email`; the subscription delivers nothing until the **confirmation click** ‚Äî runbook ¬ß4; new **`alarms_topic_arn`** output for the verify step). **Dead-man:** `AWS/Events Invocations` on the daily rule, dimension **`RuleName`** (the Surgeon corrected the spec's wrong `Rule` key ‚Äî with missing-data-as-breaching, a wrong key would have *permanently false-fired*), **86400 √ó 1** (the ONLY legal full-day shape ‚Äî CloudWatch caps period √ó evaluations at one day), `Sum < 1`, **missing = breaching** (Invocations is sparse; missing IS the dead signal); honest lag: worst case **~2 days** after the last good run; known benign quirk: a `destroy`‚Üí`apply` straddling the 06:00 UTC cron can self-clearingly false-alarm on day one. **Errors:** `AWS/Lambda Errors` (crashes/timeouts), 3600 √ó 1 `Sum >= 1`, `notBreaching`; **honest gap:** a *returned* `statusCode: 500` is a *successful* invocation ‚Äî not caught (a custom metric is the documented future refinement). Both alarms notify **recovery** too (`ok_actions`).
- **`{"mode":"smoke"}` ‚Äî the post-deploy gate for the ERR-004/005 class.** At the very top of the handler `try`: env + a Data-API connect + `SELECT version_num FROM alembic_version` vs the head this code expects (**`$ALEMBIC_HEAD`** pinned by terraform per migration; hardcoded fallback `0006_subscores`; a **staleness unit test** pins the constant to the *real* migrations head via alembic's `ScriptDirectory`). **200** match / **400** wrong head / **500** unreachable-or-never-migrated (the `error` field disambiguates); touches **no** config/LLM/S3/email and **writes nothing** (integration tests prove **zero rows across all 8 tables** with exploding fakes on all six adapter symbols); < 2 s warm. The ERR-008-compliant one-liner is [runbook ¬ß2](docs/runbooks/deploy.md) ‚Äî release-sequence step 6: **200 or you stop**.
- **`scripts/invoke.py` (the ERR-008 wrapper): explicitly decided AGAINST** ‚Äî the invocation pattern stays **procedure-only** (the runbook one-liners encode `AWS_MAX_ATTEMPTS=1` + `--cli-read-timeout`; [procedure-registry](docs/ledgers/procedure-registry.md) row updated); a wrapper waits until real ops friction proves the procedure insufficient.
- Verified: **349 unit + 45 integration green** (+5 live-key skips; full suite **394 passed, 95.21% coverage**, 85% floor), `ruff` clean, `terraform validate` clean; independent fresh-context adversarial Examiner: **CLEAN PASS, zero blocking** (its S1 `alarms_topic_arn` output + N1 400/500-boundary runbook clause applied in `a359431`; its RuleName / 86400√ó1 / unmanaged-bucket verifications all confirmed).

### Fixed

- **[ERR-009] Cold-start runs died at the first DB touch ‚Äî Aurora's scale-to-zero resume is now waited out, not fatal.** Aurora Serverless v2 resuming from pause raises `DatabaseResumingException` for ~15‚Äì30s; with `maximum_retry_attempts=0` (ERR-007, deliberate) a run that caught the cluster asleep returned 500 and stayed dead ‚Äî **three identical live kills in two days** (incl. the daily 06:00 cron), latent since v0.1 and masked only by dev activity keeping the cluster warm. New **`wait_for_db_resume`** (`db/engine.py`) probes `SELECT 1` and retries **only** the resume signal (root-cause name match walking `.orig`/`__cause__` + message-substring belt; 90s budget, 5s interval, each wait logged to CloudWatch); any other failure re-raises immediately. Wired at both repo-construction sites in the handler ‚Äî the Run-5 smoke gate's own repo (the post-deploy invocation most exposed to a paused cluster) and the shared pipeline repo ‚Äî so every mode's first query is protected. A returned-500 is invisible to the Run-5 alarms (the ADR-0029 gap) ‚Äî this fix removes the failure class rather than detecting it. *Rider (found while validating the fix):* the `jobfetcher` package log level is now set from **`$LOG_LEVEL`** (default **INFO**, safe-fallback on junk) ‚Äî Lambda's WARNING root level had been dropping **all** package INFO telemetry (reassess reports, score-shadow drift lines, these wait logs); the DB lineage log, not CloudWatch, had been the ground truth.
- **`migrations/env.py` silenced app loggers after any programmatic alembic use** ‚Äî bare `fileConfig(...)` left Python's default `disable_existing_loggers=True`, disabling every imported `jobfetcher.*` logger (surfaced as an order-dependent caplog failure in the full suite). Fixed with `disable_existing_loggers=False` + order-independence hardening in the test (`cda34b9`) ‚Äî protects all future caplog tests.

## [v0.8.0] ‚Äî 2026-07-10 ‚Äî scorer integrity: subscores + shadow totals

**The 7-factor scoring formula finally exists.** Agentic squad program **Run 4** (CRUCIAL tier ‚Äî PR-reviewed). SHADOW mode: the LLM holistic stays the product number; the weighted code-total is only logged + persisted, feeding the M7 cut-over decision.

### Added ‚Äî scorer integrity (Squad Run 4)

- **[ADR-0028] The 7-factor formula finally exists ‚Äî subscore decomposition + a SHADOW code-total.** Before: DeepSeek `v4-pro` non-determinism at temp 0 produced same-input deltas up to **27+** (live lineage data: one same-fingerprint job scored **35‚Äì88** across runs) against a near-miss band of **10** ‚Äî boundary categorizations were partly random ‚Äî and the 7-factor weights **existed nowhere**: docs said TBD, and the prompt told the LLM "the score is your holistic judgment" (the model had been freelancing the weighting since v0.1). Now `ScoreResult` carries **7 optional bounded subscores** (`core_skill_match` ¬∑ `tool_tech_alignment` ¬∑ `achievement_relevance` ¬∑ `seniority_scope` ¬∑ `ats_keyword_coverage` ¬∑ `domain_sector_fit` ¬∑ `realistic_fit`) and **`FACTOR_WEIGHTS`** (`core/scorer.py`) writes the formula down for the first time ‚Äî **.30/.20/.15/.15/.10/.05/.05** (Tarig, 2026-07-08; provenance: the original draft-PDF's ATS framework, adapted), module-load asserted `sum == 1.0`. The prompt demands **`score` FIRST** in the output (autoregressive anti-anchoring: the holistic number is emitted before any subscore tokens) + an explicit "not a mechanical average" instruction; **zero pre-existing prompt instructions changed** (Examiner AST-diffed the runtime strings).
- **SHADOW mode (Tarig's explicit choice):** `score` ‚Äî the LLM holistic ‚Äî **stays the product number everywhere** (banding, threshold, digest untouched; Examiner grep-proved no leak, and the hardened invariant test discriminates every consumer path). **`compute_code_total`** = the weighted sum with **banker's rounding** (pinned exact: 73.5‚Üí74, 72.5‚Üí72), logged as drift telemetry (INFO; **WARNING when |delta| > 20**). **Cut-over to `code_total` is an M7 decision** after ~a week of reassess data (criterion in the ADR: code-total spread ‚â§ the near-miss band ‚Üí tighten VG3 + consider cut-over).
- **Migration `0006_subscores`** ‚Äî additive nullable **`subscores` JSONB** on `score` + `score_event`; the persisted blob = `{7 factors, code_total, llm_total}` **or NULL when the LLM omitted them (never partial)**; a re-score **replaces** the current-view blob (even with NULL ‚Äî a stale breakdown must not misdescribe a new judgment; history is safe on the self-contained events). **Failure asymmetry (deliberate):** OMITTED subscores fail **open** (NULL + `"subscores_missing"` telemetry, run continues); MALFORMED/out-of-range fail **closed** (Pydantic ‚Üí the existing retry‚Üískip path).
- **Reassess becomes the variance instrument** ‚Äî the report gains **`delta_buckets`** (`0-5`/`6-10`/`11-20`/`21+`) + **`max_delta`** + **`mean_delta`** over successful re-scores; the first post-deploy reassess also doubles as the **old-vs-new-prompt drift measurement** (the Examiner's N5 residual-risk assessment, made measurable).
- Verified: **346 unit + 42 integration green** (+5 live-key skips; full suite **388 passed, 95.16% coverage**, 85% floor), `ruff` clean; independent fresh-context adversarial Examiner: **CLEAN PASS, zero blocking** (S1 test hardening + N6 boundary pin applied in `33a8ef4`).

## [v0.7.0] ‚Äî 2026-07-08 ‚Äî "the pipeline remembers": score lineage + outcome tracking + digest truthfulness

> **Release-numbering note (2026-07-10):** the digest-truthfulness work (Squad Run 3, [ADR-0027]) merged *before* the `v0.7.0` tag was cut, so it physically shipped **inside the v0.7.0 tag tree** ‚Äî it is credited here (rather than getting its own `v0.8.0` number). The subsequent scorer + ops units became `v0.8.0` / `v0.9.0` above. Live-validated **2026-07-09/10** (the truthful format is what the unattended cron now sends).

### Added ‚Äî digest truthfulness ("the digest tells the truth")

- **[ADR-0027] New/still-open split ‚Äî the daily digest stops re-presenting old matches as news.** Before: `get_scored_shortlist` was deliberately global (its own SCOPE CAVEAT), so **every above-threshold job ever scored re-appeared in every digest** ‚Äî newest buried, stale never expiring ‚Äî and reassess graduations existed only in the invoke-response JSON, never in any email. Now **"New since last digest"** leads with full cards ‚Äî NEW iff `since is None` (the first-ever digest) or `scored_at is None` (defensive) or a **fresh judgment that is actually news** (`scored_at > since` AND: a first scoring, or a graduation `previous_score < threshold <= score`) ‚Äî and everything else above threshold compacts into **"still open"** (a count line + the top-5 one-liners + "‚Ä¶and n more ‚Äî see your export", [querying.md](docs/querying.md)). A **graduation gets a green `‚Üë old‚Üínew` badge** (ADR-0023's graduations finally reach the email); a zero-new day honestly says **"no new matches since {date}"** and still sends (VG5 spirit). The split/grouping are pure render-side functions (`split_new_and_still_open`/`collapse_duplicates`), **no migration** ‚Äî `since` = the new `Repository.get_last_digest_sent_at` (`MAX(run_log.digest_sent_at)`, NULL-safe), resolved once by `notify()`.
- **Render-time duplicate collapse** ‚Äî within-JSearch duplicates measured at **17.8% of scored postings** (32/180 in 14 fingerprint groups; worst: "Data Platform Engineer @ Blackstone EIT" scored **5√ó**, 35‚Äì88, each dup paying the pro model). Same-fingerprint groups now collapse to **one card** (representative = highest score; footnote `seen n√ó ‚Äî scores lo‚Äìhi`; None/empty fingerprints never merge), each group renders in exactly **one** section (NEW iff any member is new ‚Äî a straddling group never renders twice), and the subject counts collapsed groups. **Cluster-attach dedup ‚Äî stopping the duplicate scoring *spend* at the source ‚Äî is explicitly deferred to M2** (Tarig's scope call: it belongs to the multi-source clustering redesign, ADR-0005).
- **Digest age bound** ‚Äî new **REQUIRED** `SearchSpec` knob **`digest_max_age_days`** (0‚Äì365; 0 = keep forever; sample recommends **90**): still-open matches older than N days drop out of the digest entirely ‚Äî aged by `COALESCE(posting.fetched_at, bronze_posting.fetched_at)` (the ADR-0025 reassess pattern), unknown-age rows always kept, bound-param cutoff. **Documented semantics change:** the "+N below threshold" count is now scoped to the same age window. `get_scored_shortlist` gains `since`/`max_age_days` (+ `previous_score`/`fingerprint`/`scored_at`/effective fetched-at on `ShortlistItem`, all on the Protocol); the handler threads the knob; `scripts/preview_digest.py` shows all three new states.
- **Process win (recorded honestly):** the **first auto-pilot unit** under the severity-gated policy adopted 2026-07-07 (Claude merges on a clean Examiner pass + green CI). The original spec's `previous_score`-only newness would have re-presented every old job as "New" every day in daily-only operation (jobs are scored exactly once, so `previous_score` stays NULL) ‚Äî the independent fresh-context Examiner **caught it** (a contested finding, adjudicated fix-now, fixed in `0eca0f7`) and the re-verify came back **CLEAN PASS**.
- Verified: **311 unit + 39 integration green** (+5 live-key skips; full suite **350 passed, 95.02% coverage**, 85% floor), `ruff` clean.
- **‚öÝÔ∏è Deploy sequencing (digest):** `digest_max_age_days` is **REQUIRED** ([ADR-0027]) ‚Äî the deployed S3 config must be re-pushed or **every run fails loudly** (`SearchSpec` ValidationError): **update the local config YML ‚Üí `python scripts/push_config.py` ‚Üí deploy/invoke** (done). (Registered in the [procedure registry](docs/ledgers/procedure-registry.md).)

**Live-validated on the deployed stack (2026-07-08):** migrations `0004`+`0005` applied over the Data API with the **228-score baseline backfill verified**; reassess smoke wrote **771 lineage events through the new dual-write** (the ERR-004/005-class watch item ‚Äî the inline no-RETURNING INSERT ‚Äî proven live at scale), **10 graduations standing**, deadline-bounded runs finished under the 15-min cap; one `track.py override` exercised the codebase's first **`.rowcount` over the Data API** (clean; human-override event recorded); export carries all six tables (1000 events). Operational lesson recorded as **ERR-008**: the AWS CLI's client-side retry re-invoked the long synchronous reassess (~3 passes' LLM spend) ‚Äî long invokes go async or with CLI retries disabled.

### Added ‚Äî score history + lineage

- **[ADR-0025] Append-only `score_event` log ‚Äî re-scores never erase judgments again.** Before: `save_score` upserted the `score` row in place, so every re-score destroyed the prior strengths/gaps/assessment/`scored_at` keeping only one `previous_score` generation ‚Äî and DeepSeek scores are non-reproducible even at temp 0, so overwritten history was *irrecoverable* (measured: after the single 2026-07-06 reassess, **100% of the 180 scored postings** had the original narrative already gone and the original score one invoke from permanent loss). Now migration **`0004_score_event_lineage`** adds an immutable `score_event` table (score/fit + `strengths`/`gaps`/assessment + **lineage: `scoring_model` ¬∑ `profile_hash` ¬∑ `run_id`**, `scored_at` default `now()`, indexed on cluster/run) and `save_score` **dual-writes** the (unchanged) `score` upsert + the event INSERT in **one transaction** ‚Äî either failure rolls back both. A **baseline backfill** rescues the 180 pre-0004 scores into the log (`scoring_model`/`profile_hash` = `'pre-0004'`). The event INSERT is a plain no-RETURNING/no-prefetch statement (Data-API hardening per ERR-004/005); the handler stamps every event with a **`profile_hash`** (sha256 of profile + the 3 knobs, also stored on the new nullable `profile.profile_hash`), so any score joins to the exact profile that produced it. M7 calibration + the "45‚Üí62‚Üí78" history charts (ADR-0023/0024 follow-ons) now have their data source.
- **Reassess age bound** ‚Äî `get_scored_for_reassess(max_age_days=...)` ages postings by `COALESCE(posting.fetched_at, bronze_posting.fetched_at)` (bronze is the effective live source), **includes** unknown-age rows (safe default), `0`/`None` = unbounded (query string-identical to before); new **REQUIRED** `SearchSpec` field **`reassess_max_age_days`** (0‚Äì365; 0 = no cutoff; sample recommends **45**) ‚Äî reassess stops paying LLM tokens to re-score months-old, likely-filled postings forever. The method is now declared on the `Repository` Protocol (closing a v0.4.0 omission).
- **`scripts/export.py`** ‚Äî the SQLite snapshot gains a **`score_events`** table (the score-delta/provenance index next to the flat `jobs` current view).
- Honest asymmetry (recorded in the ADR): an event's `previous_score` is what *that* `save_score` call received ‚Äî a fresh scoring writes event `previous_score=NULL` even when the score-row upsert carries an old value; deltas are recoverable from **event order** regardless.
- Verified: **272 unit + 31 integration green** (+5 live-key skips), **94.76% full-suite coverage** (85% floor), `ruff` clean; independent fresh-context adversarial + integration review ‚Äî zero blocking defects. Residual live item: watch the **first `save_score` over the Aurora Data API** in the release's live smoke (the inline INSERT is compile-verified; the Data-API path is only provable live).

### Added ‚Äî outcome tracking + human-override lineage

- **[ADR-0026] Append-only `application_event` outcome log + `scripts/track.py` ‚Äî the pipeline finally records what happens *after* the digest.** Before: `posting.status` dead-ended at `'scored'` ‚Äî "Applied / Interview / Offer / Rejected" lived nowhere, and unlike bronze (replayable), outcome data is *unreplayable*: every applied-to job not recorded was **calibration + funnel data lost forever**. Now migration **`0005_application_event`** (chains to 0004) adds an immutable outcome log (`posting_id` FK ¬∑ `status` CHECK-limited to `applied`/`interview`/`offer`/`rejected`/`withdrawn` ¬∑ `noted_at` ¬∑ optional `note`; **no backfill** ‚Äî no prior outcome data exists); re-recording is a **new event, latest wins** (append-only, deliberately no `UNIQUE`). The vocabulary lives **once** (`APPLICATION_STATUSES`, `core/models.py`) ‚Äî the table CHECK is built from it, the repository re-validates it, the CLI subcommands are generated from it, and a unit test pins the migration's frozen literals to it. Usage: `python scripts/track.py applied|interview|offer|rejected|withdrawn <posting_id> [--note]` + `find` (copy-pasteable posting_id lookup) + `events` (the outcome trail); failures are loud (stderr + exit 1, zero rows).
- **Human score overrides land in the lineage log** ‚Äî `score.score_override` (in the schema since v0 for M7 calibration) was wired to nothing; now `track.py override <posting_id> <score>` calls `Repository.set_score_override`: **ONE transaction** that UPDATEs `score.score_override` (rowcount-checked) **and APPENDs a `score_event` with `scoring_model='human-override'`** ‚Äî human corrections join the same append-only history as LLM scorings; a second override moves the column but **both events survive**. Two semantics to know: an override does **not** change `score.fit_category` (the override's derived category lives on its event ‚Äî the current view keeps the LLM's category next to your number), and `save_score`'s upsert never clears `score_override` ‚Äî **an override survives later re-scores/reassess**.
- **`scripts/export.py`** ‚Äî the flat `jobs` table gains **`score_override` ¬∑ `latest_application_status` ¬∑ `application_noted_at`** (latest event per posting; event-less postings stay NULL) and the SQLite snapshot gains an **`application_events`** table; the CSV stays the flat jobs table (now with the new columns).
- Verified: **283 unit + 36 integration green** (+5 live-key skips; full suite **319 passed, 94.74% coverage**, 85% floor), `ruff` clean; independent fresh-context adversarial Examiner: **clean pass, zero blocking** (its S-1 drift-pin finding fixed in `a6dd24a`). Residual live item: `set_score_override` is the codebase's **first `.rowcount` reliance** ‚Äî a new Aurora Data API dialect surface (ERR-004/005 lesson) ‚Äî so one `track.py override` joins the release's live smoke.

- **`employment_types` was a silent no-op with no validation** ‚Äî it was typed `list[str]` (any string accepted) **and** never actually passed to the JSearch query. Now it's an **`EmploymentType` enum** (`FULLTIME`/`PARTTIME`/`CONTRACTOR`/`INTERN`) ‚Äî a typo fails **loudly at config-load** (like `date_posted`/`remote`) ‚Äî **and** it's wired into the `/search` request, so setting it actually narrows results. `[]` still means no filter.

## [v0.3.0] ‚Äî 2026-07-03 ‚Äî user-customizable settings (no redeploy)

**Toward "fully customizable per user."** The job-seeker settings became genuinely user-owned + editable without a rebuild ‚Äî a settings change is now one command, not a deploy.

### Added
- **The 3 shortlist-strictness knobs are now user config** ‚Äî `threshold` + `hard_floor` + `near_miss_band` are all fields on the `SearchSpec` (before, only `threshold` was; floor/band were code constants), validated `hard_floor <= threshold`. ([#18])
- **[ADR-0022] Runtime config in S3** ‚Äî the two config YAMLs moved out of the Lambda zip; the handler reads them from **S3 at runtime**. **Change a setting = edit the YAML + `python scripts/push_config.py`** (validates then uploads) ‚Üí the next run uses it, **no rebuild/redeploy**. `SearchSpec`/`Profile` gain `from_yaml_text`; new `adapters/s3_config.py` (`S3ConfigStore` + `read_config_text` s3://-or-local dispatch); Terraform seeds the config to S3 with `ignore_changes` (never clobbers a runtime edit); the build no longer bundles config. ([#19])

### Fixed
- **The write-once trap** ‚Äî the handler seeded the `profile` row only on the first run, freezing the entire profile + knobs; it now **re-syncs from the config every run**, so editing a config file actually takes effect. ([#18])

## [v0.2.0] ‚Äî 2026-07-02 ‚Äî M1: pipeline hardening

**The bottleneck-driven first migration.** The P2 protocol ran the tool live on the full 18-query GCC sweep and measured three real bottlenecks (overruling the pre-drawn *M1 = CV tailoring* hypothesis): serial throughput that timed out, a single provider `503` that killed a whole run, and a gold filter loose enough to pay the pro-model to reject obvious junk ‚Äî plus AWS blind-retrying the dead run. All fixed and **re-validated live on the exact workload that failed** ([ADR-0021](docs/adr/0021-m1-pipeline-hardening.md)).

### Changed
- **Throughput (H-2):** silver dissection + scoring now run their LLM calls on a bounded **`ThreadPoolExecutor`** (default 8, `$PIPELINE_MAX_WORKERS`); **all DB writes stay on the main thread**. A **deadline guard** (`context.get_remaining_time_in_millis()` ‚àí 60s) stops starting new work before the timeout, returns `partial: true` with a `deferred` count, and skips notify so the idempotent re-run sends the digest. **Measured ~13√ó faster (~1.1‚Üí~14‚Äì15 dissections/min); a run can no longer time out.**
- **Precision (H-3):** the deterministic gold filter now requires **all** of a target title's tokens in the posting title ("Data Architect" ‚Üí `data`+`architect`), not any single shared token; the built `LlmFilterStrategy` is now selectable via `$GOLD_FILTER_STRATEGY`. **The six live junk titles (Alliances Manager, Computer Vision Engineer, ‚Ä¶) are eliminated.**
- **Lambda infra:** `memory_size` 512‚Üí**1024 MB** (CPU scales with memory for the worker threads); **`aws_lambda_function_event_invoke_config { maximum_retry_attempts = 0 }`** codifies the fix for the async zombie-retry.

### Added
- **Retry + jitter (H-1):** `OpenAICompatLlmClient.complete()` retries only transient failures (429/5xx + connection/timeout) with exponential backoff + full jitter (`LlmConfig.max_retries`, `backoff_base_s`); auth/model-not-found fail fast. `land_silver` now isolates `LlmError` symmetrically with `score_gold` ‚Äî one blip skips one posting, never the run.
- **[ADR-0021]** (M1 pipeline hardening, with the measured before/after table + the honest M3 caveat); **ERR-006** (503 no-retry crash) + **ERR-007** (async auto-retry re-fetch) in the error log; new unit tests (retry policy, failure isolation, concurrency wall-clock, deadline deferral, subset-title, strategy resolution) ‚Äî **212 unit + integration green**.

### Live validation (2026-07-02)
- Re-ran the ~132-posting backlog the pre-fix code died on: `statusCode 200`, backlog fully dissected + scored, **0 run-fatal errors** (15 dissect + 0 score failures isolated), and a **populated 21-job digest sent** ‚Äî real GCC Data-Engineer roles scored 60‚Äì95 across all six countries. (Bonus finding: the market is *not* thin; the earlier "no matches" was the tiny Oman/Architect sample.)
