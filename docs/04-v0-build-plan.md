# 04 · v0 Build Plan

> The **only fully-planned stage** (everything past v0 is planned just-in-time — [P2](00-design-philosophy.md#p2--bottleneck-driven-evolution)). This is the exhaustive apply-sequence for the minimal working core. **Do not start building it until Tarig has reviewed the docs** (the review gate). Each step has **WHY / WAIT-FOR / FAILURE-MODE**. The build itself follows safety-first: one change at a time, verify before+after, tag before risk, destructive ops need approval.

---

## v0 in one sentence
EventBridge (daily) → **one Lambda**: fetch from **one source** → land raw in **S3** + upsert rows in **Postgres** → an **LLM (DeepSeek, OpenAI-compatible API)** scores each new job against the profile → **daily email** with the scored shortlist. Terraform + Secrets Manager + tests + minimal CI. **That's all.**

**Explicitly NOT in v0:** CV tailoring · multi-source · clustering dedup (one source ⇒ only exact-id re-fetch dedup) · Step Functions · Notion · near-miss/graduation · warehouse/dbt · full observability. Each is a later migration.

## v0 contract (for [ledgers/interface-contracts](ledgers/interface-contracts.md))
- **Consumes:** `search_config` (queries/locations), candidate `profile` JSON, source API key, Bedrock access, a verified SES sender+recipient.
- **Produces:** `posting` rows (status `fetched`), `score` rows (status `scored`) in Postgres; raw payloads in S3 `raw/`; one daily email. A documented schema other migrations build on.

## Prerequisites (one-time, before the apply sequence)
1. **AWS account** in **us-east-1**, + a **DeepSeek API key in Secrets Manager** (`jobfetcher/deepseek`, never committed). The LLM transport is the **OpenAI-compatible API** ([ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md)); v0 provider = **DeepSeek** (`deepseek-v4-flash` cheap / `deepseek-v4-pro` strong) — **no new-account quota gate** (unlike Bedrock; [ERR-001](ledgers/errors.md) is now worked around, not blocking). Register at `platform.deepseek.com` (5M free signup tokens cover the backfill). · *WAIT-FOR:* a 1-call `deepseek-v4-flash` chat-completion returns a valid completion. · *FAILURE-MODE:* `401` → key/secret wrong; `model not found` → wrong model id; `429` → backoff. *(Bedrock stays a parked, config-swappable backend — flip `base_url`+`model` if its quota ever lifts.)*
2. **One job-source API key** — JSearch (RapidAPI) **or** Adzuna app id+key. (v0 uses exactly one; the second source is M2.) Store it in **Secrets Manager** (`jobfetcher/jsearch`), never env/repo.
3. **SES**: verify the sender identity and the recipient (sandbox mode is fine for a single recipient). · *FAILURE-MODE:* `MessageRejected` → identity not verified.
4. **Candidate profile**: real `profile.yml`/`profile.json` kept **locally + gitignored**; a **sanitized sample** committed so the repo is runnable by others.
5. **AWS authentication.** The **deployed pipeline uses no static keys** — each Lambda authenticates via its **IAM execution role** (AWS injects temporary creds at runtime; Terraform provisions the least-privilege role, Step 3) — nothing to configure, expire, or re-login. *Locally* (CLI/Terraform/IDE) authenticate with whichever profile you prefer: a **session login** (keyless — `aws login`, temporary STS creds cached under `~/.aws/login/cache`, re-sign-in on expiry) **or** a **non-root IAM user access key** (e.g. the `jobfetcher` profile — the AWS Toolkit extension prefers this, no session expiry). **Never create root access keys.** Third-party API keys (JSearch/SES) → Secrets Manager. · *WAIT-FOR:* `aws sts get-caller-identity --profile <your-profile>` returns your identity. · *FAILURE-MODE:* `ExpiredToken`/`InvalidClientTokenId` locally → re-auth the session or check the key; an expired *local* session never affects the *running* pipeline (that's IAM roles).

## Apply sequence

### Step 0 — Coverage probe (resolve the source on evidence, at $0)
Validate JSearch before paying. Register a RapidAPI key, subscribe to **JSearch Basic (free, 200 req/mo)**, and run the **decided query matrix** ([ADR-0010 addendum](adr/0010-job-source-jsearch.md)): **3 titles × 6 GCC countries = 18 base queries** (`Data Engineer` / `Data Platform Engineer` / `Data Architect` × `sa,ae,qa,kw,bh,om`), `date_posted=month` (30-day backfill), `remote_jobs_only=false`, capped at `max_pages_per_query` with a hard `request_budget_per_run ≤ ~70`. Config = [`config/search_config.sample.yml`](../config/search_config.sample.yml); runner = [`scripts/jsearch_probe.py`](../scripts/jsearch_probe.py) (key from **Secrets Manager** `jobfetcher/jsearch`, env-var fallback for quick tests; never committed; raw dumped to a gitignored `probe_output/`). **One sweep ≈ 40–70 requests → fits the free 200/mo** (room for 2–3 sweeps). "Backfill only for now" → inspect the real data *before* setting a daily cadence or paying for Pro.
- **Measure the 5 metrics:** (1) coverage (relevant DE postings/day per country) · (2) JD completeness (full `job_description`?) · (3) query precision (best title/location phrasing) · (4) dedup reality (`apply_options` pre-merge; do reposts reuse `job_id`?) · (5) depth (pages/query). Also confirm the **`num_pages` billing** (N requests or 1?).
- **WHY:** "is coverage good enough" is the bottleneck that justifies paying — settle it with data, not a guess. The free tier is plenty for the probe.
- **WAIT-FOR:** a clear yes/no on GCC depth + full JD text. If yes → set the daily-incremental window + subscribe **Pro ($25/mo)**. If thin → record it and reconsider (probe Adzuna / a regional board) before building.
- **FAILURE-MODE:** sparse results → widen queries (titles × cities, or add city text to the query) before concluding the source is weak; truncated JDs → that source is disqualified for scoring.

Thread the **correlation `run_id`**. Request-budget knobs (queries, page-cap, date window) live in config.
- **WHY:** bronze-first = the land-daily guarantee + replay; the adapter is the seam that makes M2 (multi-source) a one-file add.
- **WAIT-FOR:** a real call lands raw to S3 + `bronze_posting`, and ≥1 normalized `posting` in silver; re-running the same day adds no duplicate bronze rows for the same source-id.
- **FAILURE-MODE:** API shape drift → the contract catches it loudly (map the field in the adapter); quota exhausted → land what you got + log, never crash the run.
- **✅ Built (feat/step4-fetch-silver):** `SourceAdapter` port + `JSearchSourceAdapter` (productionized probe fetch) · `S3RawStore` · `clean`/`fingerprint` · `fetch_to_bronze`/`land_silver`/`ingest` ([core/ingest.py](../src/jobfetcher/core/ingest.py)). 60 unit + 11 integration (moto S3 + local Postgres) green; live JSearch→DeepSeek path tested. `bronze_id` = `posting_id` = `{source}:{source_job_id}` → exact-id dedup.

### Step 4a — Silver `Dissector` reader (build unit C-1) ✅

Delivered (`src/jobfetcher/`): the package scaffold; the `OpenAICompatLlmClient` (`LlmClient` port — `base_url`/`model`/key in config, DeepSeek default, [ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md)); the `DissectedPosting`/`Skill` contract (the silver-schema seed — de-risks Step 2); the `Dissector` ([ADR-0016](adr/0016-llm-dissection-at-silver.md)). **Accuracy built in** (plan §30): metadata-split (the LLM does *only* the free-text fields; title/location/seniority/language come from the source payload) · grounded extraction (each skill quotes a JD phrase) · `grounding_check` drops any skill whose evidence isn't in the JD · tight `must|nice|implied` defs + a worked example + temp 0.
- **WHY:** prove DeepSeek does *real, grounded work* on a real JD (beyond connectivity), and author the dissection contract early so the silver schema (Step 2) is de-risked.
- **WAIT-FOR (met):** 22 mocked unit tests green (contract, grounding, retry, parsing, error-mapping — each with a negative case); **live** on the thin `sample_sa_en` JD (zero named tools) → **0 skills, 0 hallucinated** (the guard holds); live on the detailed `sample_sa` JD → **17 grounded skills, correctly leveled, 0 dropped**. *(VG-c — a true non-English JD — deferred: all 3 current probe fixtures are English; validate at the full sweep.)*

- **Deferred ("measure, then tighten"):** a hand-labeled gold set + precision/recall once the sweep yields enough JDs; a second-pass verifier; a canonicalization dictionary; per-field stronger-model escalation. **Still to build:** storage/infra (S3, Aurora, Alembic, Terraform, the Lambda handler), the gold filter (4b), the scorer (5).

### Step 4b — Gold filter (cheap, before the LLM)
Apply the **`FilterStrategy`** over silver → mark the **gold candidate** subset (`posting.status = gold_candidate`). v0 strategy = an **LLM filter** (cheap model) judging likely-fit on the *already-dissected* structured fields (normalized title / seniority / skills / location vs the profile); **swappable by type** ([ADR-0015](adr/0015-type-replaceable-pipeline-stages.md)) — a deterministic/pass-through strategy bootstraps end-to-end tests while the LLM is quota-gated. Targets come from the user's `SearchSpec` + profile.
- **WHY:** the LLM is the expensive step — only score likely fits. Below-bar rows stay in bronze/silver for later analytics.
- **WAIT-FOR:** an obviously-irrelevant posting (wrong location/title) is **excluded** from gold while a matching one is included.
- **FAILURE-MODE:** filter too aggressive → real fits dropped before scoring. Keep it *coarse* (the LLM does the fine judgment); log filtered counts.
- **✅ Built (feat/gold-filter, build unit Unit A):** the `Profile` contract ([core/profile.py](../src/jobfetcher/core/profile.py) — `skills`/`certifications`/`projects`/`preferences`, `extra="forbid"`, `from_yaml`/`from_jsonb` fail loud; sample [config/profile.sample.yml](../config/profile.sample.yml), real → gitignored `config/profile.local.yml`); the `FilterStrategy` port + `FilterError` ([core/ports.py](../src/jobfetcher/core/ports.py)); two strategies — `DeterministicFilterStrategy` ([adapters/filter_deterministic.py](../src/jobfetcher/adapters/filter_deterministic.py)) and `LlmFilterStrategy` ([adapters/filter_llm.py](../src/jobfetcher/adapters/filter_llm.py): `deepseek-v4-flash`, temp 0, structured `likely_fit` JSON, one retry, **fails open** on error); 5 new `Repository` methods (`get_profile`, `get_silver_postings`, `mark_gold_candidate`, `upsert_cluster`, `set_posting_cluster`); and `apply_gold_filter(spec, profile, *, strategy, repo, source)` ([core/ingest.py](../src/jobfetcher/core/ingest.py)) → silver → per-posting `strategy.filter(...)` → fits get a 1:1 cluster + `status='gold_candidate'`; returns `{silver, gold, dropped}`; **fails open only on `FilterError`** (a real strategy bug stays loud). 105 unit + 16 integration green; `ruff` clean.
- **⚠️ Logged, defended deviation — v0 default strategy.** The step text above said *"v0 strategy = an **LLM filter**"*; the build makes the **v0 default = `DeterministicFilterStrategy`** (coarse title/location/avoid-keyword match, no LLM). **Why (P1):** at 10–30 jobs/day an LLM gold-filter is largely *redundant with the Scorer* (the next stage already does fine LLM judgment), so paying for it twice fails minimalism. The `LlmFilterStrategy` *is* built and **config-selectable** for scale — the seam stays open, the cost doesn't land. This is a logged, defended choice, not a silent change.
- **🔧 Resolution — the "score keys on `cluster_id` but v0 has no clustering" gap.** Each fit gets a **1:1 cluster** (`cluster_id = posting_id`) + `posting.cluster_id` set, so the Scorer (Step 5) can key on `cluster_id` today with zero special-casing. **Real clustering (cluster-and-surface dedup) is M2** — this 1:1 row is the cheap forward-compatible stand-in.
- **Known minor deferrals** (for the Scorer/handler authors): (a) `mark_gold_candidate` / `set_posting_cluster` are **silent no-ops on a missing `posting_id`** — unreachable in v0 (the id always comes from `get_silver_postings`); (b) `get_silver_postings`' `limit` is **unordered** and **unused in v0**; (c) the deterministic filter is **intentionally coarse/permissive** (spec-compliant — the Scorer does the fine judgment); (d) integration tests: prefer **`127.0.0.1`** over `localhost` in `JOBFETCHER_DB_URL` (IPv6 `::1` flakiness — dev-box/runbook note, not a code defect).

### Step 5 — Scorer (LLM)
Implement scoring **over the gold candidates only**: load profile, build the 7-factor ATS system prompt, call the **LLM via the OpenAI-compatible API** with the **config-selected model** ([ADR-0012](adr/0012-model-agnostic-llm.md) · [ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md); v0 = `deepseek-v4-pro`) at **temperature 0**, with prompt-based structured-output enforcement, parse into the score contract (`score`, `fit_category`, `strengths`, `gaps`, `strategic_assessment`, `poster_type`, `legitimacy_verified`; `skills`/`sector` are read from the silver dissection on `posting` — [ADR-0016](adr/0016-llm-dissection-at-silver.md) — not re-extracted), write `score` rows. Apply the **single config threshold** (default **60**) + hard-floor **50** + near-miss band **10** — **read from the per-user `profile` config at runtime, never hardcoded.** This one threshold gates the email shortlist now (and CV writing from M1). Stamp the active threshold onto each run's records.
- **WHY:** scoring is the core value; explainability is the value, not just the number.
- **WAIT-FOR:** the same JD+profile scored twice lands within ±3 points (determinism check). **→ softened to best-effort — see the VG3 decision below.**
- **FAILURE-MODE:** `401` / `model not found` → key or model id; `429` → backoff; malformed JSON → tighten the output schema / add a single retry.
- **✅ Built (feat/scorer, build unit Unit B):** the `ScoreResult` contract ([core/models.py](../src/jobfetcher/core/models.py) — score 0–100, strengths, gaps, strategic_assessment, poster_type, legitimacy_verified); the `Scorer` ([core/scorer.py](../src/jobfetcher/core/scorer.py)) — mirrors the `Dissector`: a grounded, **temperature-0**, JSON-only **7-factor ATS** `SCORING_SYSTEM_PROMPT` (core-skill match · tool/tech alignment · achievement relevance · seniority/scope · ATS-keyword · formatting/clarity · realistic fit), one parse/validate retry, typed `ScorerError`; reads the **silver dissection** (skills/sector/title/seniority/location) + the `Profile` — does **not** re-extract ([ADR-0016](adr/0016-llm-dissection-at-silver.md)); scoring model `deepseek-v4-pro` selected by the injected `LlmClient`. 3 new `Repository` methods — `get_gold_candidates` (`status='gold_candidate'` → `(posting_id, cluster_id, DissectedPosting)`), `save_score` (idempotent upsert keyed on `cluster_id`, carries `previous_score`), `mark_scored`. The orchestrator [core/ingest.py](../src/jobfetcher/core/ingest.py): `derive_fit_category` (band routing) + `score_gold` (load profile → score each gold candidate → save → mark scored; a scoring/LLM failure is logged + skipped, a DB error stays loud; returns `{gold, scored, surfaced, failed}`). **129 unit + integration green; `ruff` clean.**
- **🔧 `fit_category` band derivation (from the runtime profile knobs, no new knob).** `fit_category` is **not** produced by the LLM — `derive_fit_category(score, *, threshold, hard_floor, near_miss_band)` maps the numeric score against the per-user bands: `score >= threshold` → **`strong_fit`** (the surfaced shortlist); else `score >= threshold - near_miss_band` → **`near_miss`** (the watch band); else `score >= hard_floor` → **`stretch`** (a real-but-distant fit kept for analytics); else **`misaligned`** (below floor, analytics only). This is the 4-bucket `fit_category` the ERD names — derived purely from the configured bands.
- **🔒 `UNIQUE(score.cluster_id)` — migration 0002.** The 1:1 score↔cluster invariant (Unit A gives each fit a 1:1 cluster) is made a **DB guarantee** via [migration 0002](../migrations/versions/0002_score_cluster_unique.py) (down_revision 0001), which lets `save_score` use the idiomatic `on_conflict_do_update` for idempotent re-runs.
- **⚙️ VG8 — threshold is runtime-config.** The gate (`score >= threshold`) and the `fit_category` bands read `threshold`/`hard_floor`/`near_miss_band` from the **`profile` row at runtime** — changing the value (no code change/redeploy) changes which jobs surface and how they bucket.
- **⚠️ KEY DECISION — VG3 (determinism) softened to BEST-EFFORT.** The step's WAIT-FOR / VG3 wanted "same JD+profile scored twice within ±3 points." Reality: **`deepseek-v4-pro` is non-deterministic even at temperature 0** (MoE routing / FP variance) — observed deltas of 0, 6, 10, 13, 14, **27** across runs. The code is correct: it sends temp 0 with a stable prompt (the independent verifier confirmed — temp 0 is the guaranteed invariant). **Decision (Tarig): accept best-effort.** The v0 score is a **triage signal**, not a precise number; the live test now asserts (a) temp 0 *is* configured and (b) a generous best-effort sanity band (~±20 — still catches a temp-≠-0 regression). A borderline job can occasionally drift across the threshold; **precise stability + calibration is deferred to M7** (the `score_override` accuracy loop). **Rejected alternatives:** switch the scoring model (deviates from ADR-0017's v0=DeepSeek, adds an eval burden); average N samples (3× cost/latency — anti-P1 at 10–30 jobs/day). Recorded in [decisions-locked](ledgers/decisions-locked.md); no standalone ADR (a decisions-locked row + this note suffice, matching the threshold amendment in journal §12).
- **⏳ FOLLOW-UP (pending — apply on the Step 7 branch).** The provisional **~±20** live sanity bound is *itself* too tight: a later run drifted **27** (> 20), so the live determinism test ([`tests/test_live_scorer.py`](../tests/test_live_scorer.py)) can **intermittently fail** — not a code or product bug, just the bound out-pacing the model's real drift. **Planned refinement (not yet applied):** *drop the hard delta assertion entirely*; keep only the guaranteed invariant — assert temp 0 **is** sent — and **log** the run-to-run delta as an observation (assert what we control; observe what we don't). **Not blocking:** the live test runs only locally with a DeepSeek key; CI skips it. Cross-referenced in [journal §24](01-session-decision-journal.md).

### Step 6 — Notifier (SES daily digest) ✅ built
Render a clean daily email: top matches (≥ threshold) with score, one-line why, and the apply link; a short "below threshold" count. Send via SES.
- **WHY:** email is the v0 surface — morning triage in 60 seconds.
- **WAIT-FOR:** a real digest email arrives with correct content and links.
- **FAILURE-MODE:** `MessageRejected` → unverified identity (sandbox) — verify recipient.
- **🧱 What was built.** The **`Notifier`** port + **`NotifierError`** (`core/ports.py` — ADR-0015's previously-undefined port, now defined: `send(*, subject, html_body, text_body, recipients) -> message_id`); **`SesNotifier`** (`adapters/ses_notifier.py` — boto3 SES `send_email`, HTML + plaintext UTF-8, lazy/injectable client mirroring `S3RawStore`, sender from `$SES_SENDER`, SES errors → `NotifierError`); a dependency-free **`render_digest`** (`core/notifier.py`); **`Repository.get_scored_shortlist(*, threshold)`** (JOIN `score`↔`posting` on `cluster_id` → surfaced `ShortlistItem`s ordered by score DESC + a below-threshold count); and the **`notify`** orchestrator (`core/ingest.py` — load profile threshold → `get_scored_shortlist` → `render_digest` → `send`; a send failure **raises loud**, email being the v0 surface).
- **✅ VG5 covered.** Matches → a digest of `score >= threshold` (score · normalized_title · company · one-line why · apply link) + a "+N below threshold" footer; **zero matches → a valid "no matches today" email** (not a crash, not silence).
- **🔒 apply_url scheme allowlist (email-injection guard).** All user/JSearch-sourced text is HTML-escaped, and the apply-link `href` is **scheme-allowlisted to http/https** — a `javascript:`/`data:` URL renders "(no link)" instead. Verified un-bypassable.
- **⚙️ Single threshold authority.** The threshold is resolved **once** in `notify()` and passed into `get_scored_shortlist` — one authority, no drift between the query gate and the render gate.
- **🌍 Terraform (additive, not yet applied).** `var.recipient_email` + a `RECIPIENT_EMAIL` Lambda env var + an output — the SES sandbox requires verifying the recipient identity.
- **⏭️ Send-idempotency deferred to Step 7.** "At most one email per run date" is the once-daily handler's concern (a run-date guard), **not** the Notifier's — the Notifier sends what it's told; Step 7 owns the guard (VG4).

### Step 7 — Single Lambda handler ✅ built
Wire fetch → score → notify in one handler, idempotent for a given run date (re-running the same day doesn't double-write or double-email). Emit structured logs prefixed with the correlation `run_id`.
- **WHY:** one Lambda is the minimal orchestration; idempotency is a foundational reliability property (and a migratability requirement).
- **WAIT-FOR:** two runs for the same date produce the same DB state and at most one email.
- **FAILURE-MODE:** partial failure mid-run → the run is resumable on the next invocation (status field drives it).
- **🧱 What was built (feat/handler).** `handler(event, context)` ([handlers/pipeline.py](../src/jobfetcher/handlers/pipeline.py)) wires `ingest → apply_gold_filter → score_gold → notify` in one Lambda; **seeds the `profile` row if missing**; threads the correlation **`run_id`** through structured logs; **resolves the DB engine from env** — local `$JOBFETCHER_DB_URL` **vs** the Aurora Data API URL built from `$DB_CLUSTER_ARN` / `$DB_SECRET_ARN` / `$DB_NAME` — and the config paths (`$SEARCH_CONFIG_PATH` / `$PROFILE_PATH`); returns a `{statusCode, run_id, …stage counts}` summary (a stage failure → **`500`** so the next invocation retries).
- **🔒 `run_log` send-once guard — migration 0003.** New `run_log` table ([db/tables.py](../src/jobfetcher/db/tables.py)) + [migration 0003](../migrations/versions/0003_run_log_send_guard.py) (down_revision 0002): PK `(run_date, user_id)` + `run_id` + `digest_sent_at`. Three new `Repository` methods — `upsert_profile`, `was_digest_sent`, `mark_digest_sent`. `notify` checks the guard **before** sending and marks it only **after** a successful send.
- **♻️ Resumability (VG4).** ingest / gold / score are idempotent via upserts → a re-run reaches identical DB state; `notify` is guarded to send **at most once per `(run_date, user_id)`**. A stage failure → `500` → the next scheduled run resumes (no duplicate rows, ≤1 email) — both the same-date double-run and the crash-mid-run cases are asserted by integration tests.
- **🌍 Terraform (additive).** [lambda.tf](../terraform/lambda.tf) handler entry-point → `jobfetcher.handlers.pipeline.handler`; added `SEARCH_CONFIG_PATH` / `PROFILE_PATH` env; **fixed `DATA_BUCKET` → `JOBFETCHER_DATA_BUCKET`** (the name the code actually reads — a latent Step-10 runtime crash, fixed here). **Packaging/zip + the real apply are deferred to Step 10.**
- **⚠️ KEY DECISION — VG4 "≤1 email" is at-least-once (the dual-write window).** `notify` sends via SES (external) **then** writes the `run_log` guard — two writes that **can't be atomic** (SES isn't in the DB transaction). In a rare crash window — SES succeeds but the immediately-following `mark_digest_sent` fails — the next run re-sends (a duplicate digest). The build chose **at-least-once on the email** (send, then record) as the **safer default**: a rare duplicate is far better than a silently-missed digest. **VG4 holds for the normal cases** (a same-date double-run sends ≤1; a crash mid-run resumes with no duplicate rows + ≤1 email — both asserted); the residual window is **acknowledged, not hidden**. **Rejected:** guard-*before*-send (risks a *lost* digest if the send then fails — worse); a transactional outbox (overkill for a daily cron at 10–30 jobs/day) — the documented scale-up if it ever matters. (Same spirit as the VG3 best-effort call: state the limitation plainly.) Recorded in [decisions-locked](ledgers/decisions-locked.md).

## v0 Validation Gate (behavioral + negative — a presence check is *no gate*)

| ID | Positive case | Negative case |
|---|---|---|
| **VG1 — Ingestion** | A real source call persists ≥1 normalized posting to Postgres *and* its raw JSON to S3. | Feed a malformed API payload → the contract rejects it and logs; it is **not** silently persisted. |
| **VG2 — Scoring is behavioral** | Score a known JD+profile → score in the expected band, with non-empty `strengths`/`gaps`/`strategic_assessment`. | Score a clearly-misaligned JD → it lands **below** the floor (proves the scorer discriminates, not just returns 200). |
| **VG3 — Determinism (best-effort)** | Same JD+profile scored twice → lands within a **generous sanity band (~±20)** *and* temp 0 is asserted-configured. (`deepseek-v4-pro` is non-deterministic even at temp 0 — ±3 is not achievable; v0 score is a triage signal. Precise stability/calibration → **M7**. See the Step 5 VG3 decision.) | Configure a non-zero temperature → the determinism assertion / temp-0 check fails (catches a temp-≠-0 regression). |
| **VG4 — Idempotency** | Two handler runs for the same date → identical DB state, ≤1 email (ingest/gold/score idempotent via upserts; `notify` guarded once-per-`(run_date, user_id)` by the `run_log` table). **Email is at-least-once:** SES (external) + the `run_log` write can't be atomic, so the rare crash *between* a successful send and `mark_digest_sent` re-sends rather than drops — the safer default; transactional outbox = the scale-up (see the Step 7 VG4 decision). | Kill the handler mid-run, re-invoke → it resumes; no duplicate rows, ≤1 email. |
| **VG5 — Notification** | Daily email arrives with correct matches, scores, and working apply links. | Zero matches above threshold → a valid "no matches today" email (not a crash, not silence). |
| **VG6 — Teardown** | `terraform destroy` removes everything; bill returns to ~$0. | (N/A — destroy is the negative of apply.) |
| **VG7 — Secrets hygiene** | Secret scan passes; no secret in the repo or logs. | Plant a fake key in a staged file → pre-commit/secret-scan **blocks** it. |
| **VG8 — Threshold is config** | Change `threshold` in the per-user config (no code change/redeploy) → the next run surfaces a different set of jobs accordingly. | Set threshold above every score → the run produces a valid "no matches" email; set it to 0 → all scored jobs surface. (Proves the gate reads config at runtime, not a hardcoded constant.) |

All VGs must pass before tagging `v0.1.0`. Any failure → log an `ERR-NNN` in [ledgers/errors.md](ledgers/errors.md) (root cause + prevention + **detection**) before proceeding.
