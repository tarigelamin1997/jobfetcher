**So-what.** This session is the interrogation + design. The deliverable is this doc set. The old artifacts were fully absorbed (their substance lives on where it earns its place) and then deleted for a clean slate.

## 2. Goals & what "better" means (Section A)

- **Nothing failed ŌĆö "it was fine, want it better."** The restart is a smarter second pass, not a rescue. **Truly greenfield** (old ideas as inspiration only).

## 3. The real job-search situation (Section B)

- **Active-but-selective**, urgency **in weeks**. *So-what:* he'll actually use this soon ŌåÆ it must be genuinely good, and value must arrive fast. This created the first "full system but fast" tension, resolved by **MVP-first sequencing**.

- **Auto-CV = core value.** *So-what:* CV tailoring stays first-class.

- **DE / Data Platform / Data Architect only** (no spread to Analytics-Eng/Cloud/ML). **Riyadh ŌåÆ GCC ŌåÆ open to relocate; NOT remote-global** (on-site oriented). **Profile accurate as-is** (use it unchanged as scoring source of truth). **English only.** Old sensitivities still hold (site offline; "Cansa Group" naming; honesty rules).

## 7. Pipeline internals (Sections HŌĆōK, M, NŌĆōS)

- **Data:** multi-source via **official APIs only** (JSearch + Adzuna to start), no scraping ŌåÆ a **pluggable source-adapter** layer. Volume **modest (10ŌĆō30/day)** ŌåÆ Bedrock cost trivial; dedup is about *quality* not scale.
- **Scoring:** keep the **7-factor ATS framework** (tune weights), **explainability is critical** (strengths/gaps/strategic assessment), a **lightweight calibration loop** (capture score corrections ŌåÆ tune prompt + a scoring-accuracy SLO), keep the scam-gate + poster-type label.
  - **Threshold (amended post-design):** initially set at 75 with an implicit two-gate idea; **changed to a *single* user-configurable `threshold` (default 60), read at runtime, gating *both* the shortlist and CV writing** (floor 50, near-miss 10). *Why:* Tarig wants the CV-writing bar to be one knob each user tunes frequently without a redeploy, and a single gate is simpler/more defensible than two. The active threshold is stamped per run for measurement. *Rejected:* two separate thresholds (shortlist vs CV) ŌĆö unnecessary complexity for one user. (See plan ┬¦12.)
- **CV:** **drop LibreOffice-in-Lambda** (the old #1 reliability risk) ŌåÆ a deterministic, dependency-light renderer (one content model ŌåÆ DOCX via python-docx + PDF via a pure-Python/HTML path). Keep the *tarig-cv* template as the refined base. **One master CV.** **Strict honesty + human-review gate** (CV is a draft you approve; the review gate doubles as calibration capture).
- **UI:** **email + Notion both** first-class; status tracking in Notion; a BI dashboard over the marts is *designed-for, build-later*.
- **Near-miss/graduation:** keep the full watchŌåÆre-scoreŌåÆgraduate loop. **Observability:** initially "full suite," later **right-sized** (┬¦8). **Orchestration:** Step Functions (AWS-native) ŌĆö but it became a *migration* (M3), not a v0 component (┬¦9).

## 8. The architecture synthesis, then four deep pivots

I synthesized a **two-plane architecture** ŌĆö an **operational plane** (the daily serverless tool) cleanly separated from an **analytical plane** (DE-depth: warehouse + dbt). *Why:* lets us push hard on the DE-depth prime directive without diluting the serverless story; realizes the medallion as a *real dbt-modeled warehouse*, not just S3 folders. Tarig approved it "with tweaks," then drove four deep pivots:

### 8a. Warehouse: Snowflake vs Databricks vs Postgres
*Decision:* **Snowflake** as the (eventual, conditional) warehouse ŌĆö **not** Databricks. *Why:* the assessment's flagged gap is **SQL/warehouse modeling + dbt**, which is Snowflake's lane; at 10ŌĆō30 rows/day the warehouse is a *portfolio/skill* choice, not a data need, and Snowflake is the cheapest/most-reliable/cleanest-with-dbt option, with strong KSA/GCC demand. *Rejected:* **Databricks** ŌĆö Spark-on-tiny-data is unconvincing and a weaker fit; the Spark/Delta signal belongs in the sibling **OrderFlow** project where volume justifies it. *Crucial reframe:* since we already run **Postgres**, the entire medallion could live there for $0 ŌĆö so the warehouse choice is purely about signal, which later (┬¦9 tool-minimalism) made **Snowflake conditional** and **Postgres+dbt the default**. See [ADR-0004](adr/0004-warehouse-strategy.md).

### 8b. Operational store: Postgres + (deferred) Debezium CDC, replacing DynamoDB+Streams
*Decision:* **managed PostgreSQL** as the operational store. *Why:* the data is inherently relational (clusters ŌåÆ postings ŌåÆ scores ŌåÆ applications); a relational DB is the *right-tool fit*, and it mirrors Tarig's real DE expertise. *Rejected:* DynamoDB + Streams ŌĆö NoSQL for relational data was the *weaker* original choice. **Debezium CDC** was adopted then **deferred**: at this volume, nightly batch ŌåÆ warehouse + incremental dbt is right-sized; Debezium becomes the *documented scale-up path* ("we'll need it later"), and the built CDC showcase lives in OrderFlow. See [ADR-0003](adr/0003-postgres-over-dynamodb.md), [ADR-0009](adr/0009-batch-not-debezium-v0.md).

### 8c. Deduplication: cluster-and-surface, never hide
*The 99.99%-accuracy question.* Tarig wanted near-perfect dedup across multiple APIs. *Reframe (honest):* a literal 99.99% guarantee on fuzzy job-text matching isn't honest. There are two error types ŌĆö a **missed duplicate** (trivial cost: one extra cheap score) and a **wrong merge** (the dangerous one: a real job gets hidden). *Decision (Tarig's, and it overrides the old "pick one canonical + hide duplicates"):* **never silently hide a posting.** Instead **cluster** suspected-same postings and **surface the whole group** with every platform's apply-link + a "suspected same as X/Y/Z" note; the **user decides** whether to apply via one or several (hiring teams sometimes favor one platform's pipeline). Uncertain clusters go to a dedicated **Suspected-Duplicates** surface to confirm/split. *Engineering:* precision-first + fail-safe + **measured precision/recall** (entity resolution with real numbers ŌĆö a strong DE signal); JD-body embeddings (pgvector) + apply-URL match + company canonicalization + time-window scoping; **score + tailor a CV once per cluster** (identical content) but show all links. See [ADR-0005](adr/0005-dedup-cluster-and-surface.md).

### 8d. Distribution: self-hosted / open-source (not SaaS)
*Decision:* **self-hosted / open-source.** *Why:* the goal is a job-search tool + a portfolio, **not a company.** Self-hosting makes the IaC + reproducibility *itself* the portfolio value, with zero ongoing liability. *Rejected:* **centralized SaaS** ŌĆö it would mean owning others' cost, auth, billing, support, multi-tenancy, and **other people's CVs/PII** (real legal/privacy liability) plus job-data ToS exposure: a startup, not a portfolio piece, and a distraction from landing a job. Multi-user stays a documented future pivot. See [ADR-0007](adr/0007-self-hosted-distribution.md).

## 9. The honesty audit, the evolutionary reframe, and the governing principles

**Honesty audit (Tarig's challenge: "can you defend this architecture, or are we spinning service names?").** I audited every component against a **defensibility rubric** (now in [00-design-philosophy](00-design-philosophy.md)). Outcome: most pieces are defensible as right-tool-fit; **two were flagged** ŌĆö Debezium-CDC (right-sized to batch, deferred) and the full SLO/alarm suite (**right-sized** to a few real alarms + documented SLOs). Adopted dial: **BALANCED** ŌĆö simplest defensible option by default, 2ŌĆō3 labeled showcases (warehouse/dbt + measured entity-resolution).

**Evolutionary architecture reframe (Tarig ŌĆö the single biggest idea).** Don't build the full system at once. Build the **minimal working core (v0)**, then evolve through a **sequence of deliberate, observable migrations ŌĆö each a clean GitHub release.** *Why it's the strongest framing:* controlled architectural *evolution* (an ADR + migration guide per step) is a rare senior/staff portfolio signal; it makes every piece defensible *by construction* (capability arrives exactly when justified); and a leaner v0 yields *more, better* migrations ŌĆö the migrations ARE the showcase. *So-what:* "full system" becomes the *destination reached via migrations*, fully reconciling "keep it all" with "absolute minimalism."

**The roadmap is directional, not fixed (Tarig).** You can't draw the full sequence before shipping ŌĆö each stage's *implementation* is the bottleneck that reveals the next capability. So we commit firmly to only: v0, the migratable architecture, and release discipline. The rest is a living hypothesis. This refined "plan everything then build" ŌåÆ **"plan the foundation + current stage fully; plan each migration just-in-time."**

**Governing principles P1/P2 (Tarig).** **P1 absolute minimalism** (minimal complexity for the present problem; complexity is entropic ŌåÆ resist it) + **P2 bottleneck-driven evolution** (solve the top bottleneck with the minimal migration; the migration-decision protocol). Together: *add the minimum, only to break the biggest real bottleneck.*

**Tool-minimalism wins (Tarig).** Only build what a real **tool** bottleneck justifies; DE-depth is the *tiebreaker*, not a license to add. *So-what ŌĆö this revised an earlier "lock":* **Snowflake became conditional**, with **Postgres + dbt as the default** analytics; Step Functions survives only because it's *earned* by real Lambda complexity (M3). DE-depth is still fully served, minimally and honestly (Postgres + dbt modeling/tests/lineage + measured entity resolution + data contracts + the evolutionary-architecture story itself).

## 10. v0 boundary & the roadmap

- **v0 (locked) = the irreducible core:** EventBridge (daily) ŌåÆ **one Lambda**: fetch from **one source** ŌåÆ raw to **S3** + rows to **Postgres** ŌåÆ **Bedrock** score vs profile ŌåÆ **daily email digest**. Terraform + Secrets Manager + tests + minimal CI. *No CV, no Notion, no Step Functions, no dedup-clustering, no warehouse.* It works (you wake to a scored shortlist); everything else is a migration.
- **CV = M1** (first migration), **email-only** for v0, **single Lambda** for v0 ŌĆö chosen *leanest*, because a leaner v0 creates more (better) migration stories.
- **Roadmap (directional):** v0 ŌåÆ M1 CV ŌåÆ M2 multi-source+clustering-dedup ŌåÆ M3 Step Functions ŌåÆ M4 Notion+near-miss ŌåÆ M5 dbt marts on Postgres ŌåÆ M6 skill/sector intel ŌåÆ M7 observability+calibration ŌåÆ M8 CI/CD+README/diagram/demo (= v1.0.0). Future: Debezium CDC, multi-user, feedback hub, BI dashboard, Snowflake (conditional). **Semver:** v0.x per migration ŌåÆ v1.0.0 at M8. Full detail + the bottleneck protocol in [03-roadmap](03-roadmap.md).

## 12. AWS identity ŌĆö from keyless root to a non-root dev user
**Context.** Setting up local AWS access surfaced that the working profile (`samareltayeb`, the account **root**) had **no static access keys** ŌĆö it authenticates via a **session login** (temporary STS creds ŌĆö access key + secret + *session token* ŌĆö cached under `~/.aws/login/cache/`, refreshed at sign-in). Tarig, reasonably, read "no key/secret" as "no auth" and thought we had to create keys. That became a teaching thread: AWS auth is *temporary credentials*, not necessarily stored secrets.
**Discussion / what we weighed.** Tarig wanted access keys. The hard line: **never create *root* access keys** ŌĆö a leak = total, unrecoverable account takeover (AWS's #1 anti-pattern). The session-login worked for the CLI, but the **AWS Toolkit extension** kept dropping (its connection expired separately). Options: (a) keep keyless root session login; (b) create a **non-root IAM user** with its own access key. We chose (b) ŌĆö it is the access-key method Tarig wanted, done right, *and* AWS's own "stop using root" best practice. (An interim step first made keyless `samareltayeb` the persistent default via user env vars ŌĆö plan ┬¦18 ŌĆö before the IAM user replaced it.)
**Decision.** Created the non-root IAM admin user **`jobfetcher-dev`** with an access key; made it the **default for all local dev** (`AWS_PROFILE=jobfetcher` persistent + `[default]` mirrors the same key, so every resolution path ŌĆö flags, env, new shell ŌĆö lands on `ŌĆ”:user/jobfetcher-dev`); root retained only for rare root-only ops via an explicit `--profile samareltayeb`. Codified as a durable standard in CLAUDE.md: **humans ŌåÆ `jobfetcher-dev`; machines (CI / Lambda runtime) ŌåÆ their own least-privilege IAM roles, never the personal key.** All keys stay out of the repo and out of any cloud routine. Verified post-VS-Code-restart (Toolkit status bar showed `profile:jobfetcher`). ŌåÆ plan ┬¦18ŌĆō┬¦19, [decisions-locked](ledgers/decisions-locked.md) (Security), CLAUDE.md. *(The two-tier auth model ŌĆö local session/IAM-user temp creds vs Lambda execution roles in production ŌĆö is documented in the build-plan Prerequisites; no static key exists anywhere in the deployed system.)*

## 13. The Bedrock quota wall (ERR-001) and the model-agnostic response
**Context.** "Can't use Anthropic models." A read-only investigation (account `198592435375`, us-east-1) found *two* distinct failures: **(a)** the base model ids (`anthropic.claude-ŌĆ”`) reject on-demand ŌåÆ the Claude 4.x models are **inference-profile-only** (the `us.anthropic.ŌĆ”` cross-region id works); **(b)** ŌĆö the real wall ŌĆö the account's **per-day Bedrock token quota is `0` and `Adjustable = False` for *every* model.** Ruled out billing (valid card + credits) and IAM (root *and* the admin user both hit it). It is a **new-account maturity gate**, independent of billing.
**Discussion.** Tarig: "Kimi K2 Thinking works fine ŌĆö use it instead of Anthropic, and make switching models never a bottleneck." So we locked **model-agnostic LLM** ([ADR-0012](adr/0012-model-agnostic-llm.md)): one `LlmClient` port over Bedrock **Converse**, model id in config **per task** (`scoring_model`, `extraction_model`, ŌĆ”), structured output via prompt + Pydantic (portable, not provider-specific JSON modes). **But** verification cut against the easy story: `moonshot.kimi-k2-thinking` is **ACTIVE + ON_DEMAND** (no inference-profile gotcha ŌĆö a real simplicity win) yet its daily-token quota *also* reads `0` ŌĆö the wall is **account-wide, not Anthropic-specific.** A 1-token `converse` test confirmed it: **Kimi throttles via the `bedrock-runtime` API exactly like Anthropic.** Where Kimi "worked" for Tarig was the **console playground** (separate limits), not the API the pipeline uses. Re-tested **2026-06-23** (Ōēł1 week on) ŌĆö still `ThrottlingException`; we refused to mark it resolved on a console anecdote.
**Decision.** **Kimi K2 Thinking is the chosen model** (config-swappable; no longer waiting on Anthropic) ŌĆö but choosing Kimi settles the *model*, not the *quota*. ERR-001 remains the one blocker on the scoring path. Filed **AWS Support case `178220019100382`** to raise the new-account quota (the only deterministic lever ŌĆö quotas are non-adjustable via the console); no fixed lift timeline; we **re-test in-session** each resume (a cloud cron was declined ŌĆö it would require putting the personal AWS key in a cloud env, against the security model). ŌåÆ [ERR-001](ledgers/errors.md), [ADR-0012](adr/0012-model-agnostic-llm.md), plan ┬¦16ŌĆō┬¦17, ┬¦24ŌĆō┬¦25. **The wall held for weeks ŌĆö ┬¦18 is how we got off it (we stopped waiting and routed around it to DeepSeek).**

## 14. Ingestion ŌĆö query strategy, the JSearch reality, and the user-input contract
**Context.** The first concrete ingestion work: how to query JSearch within a rationed quota, proven on real data ŌĆö and how the user specifies *what* to search. The principle that fell out: **free tier = learn the strategy; Pro = run it.**
**Discussion / findings (all from the live API).**
- **Query matrix:** 3 core titles (`Data Engineer`, `Data Platform Engineer`, `Data Architect`) ├Ś 6 GCC countries (`sa, ae, qa, kw, bh, om`) = **18 base queries**; `date_posted=month` as a one-time **backfill** window (explicitly distinct from a future daily-incremental window). Free Basic = 200 req/mo; one all-GCC sweep Ōēł 40ŌĆō70 req ŌåÆ **fits free** (this sweep *is* the probe); Pro (10k/mo, $25) deferred until the probe gives evidence. "Backfill only for now."
- **`language=en` is mandatory:** without it, `country=sa` returns metadata in **Arabic** *and* leaves `job_city`/`job_country` + the UTC timestamp **null**; with it they populate cleanly. A real probe call confirmed full JD text arrives (Ōēł1,230 chars) ŌĆö the critical "not truncated" check that disqualifies a source.
- **Location model (honest):** the **`country` query param is the authoritative geo scope** (per-record `job_country` is unreliable / often null); `job_location` is a denormalized city string; **there is no employer-HQ location** ŌĆö JSearch gives only the job's location. City/state are therefore **gold filters**, not query params (pull broadly by country ŌåÆ filter cities in gold ŌåÆ every city still lands in bronze for analytics).
- **Secrets:** adopt **AWS Secrets Manager** for all secrets ŌĆö convention `jobfetcher/<service>` (e.g. `jobfetcher/jsearch`), JSON value, read by scripts *and* Lambdas; one secret serves local probe + production (store-once, use-everywhere). The key is never committed.
**Decision.** **Search targeting = a validated, fully-explicit per-user `SearchSpec`** (Pydantic; *every* field required ŌĆö "nothing taken for granted as input"): `job_titles` ŌåÆ query text, `countries` ŌåÆ the `country` param, `cities`/`states` ŌåÆ gold filters, plus knobs (`date_posted`, `language`, `employment_types`, `remote`, `threshold`) + budget (`max_pages_per_query`, `request_budget_per_run`). Intake = a gitignored config file the user fills, **validated loudly at load** (a missing field fails ŌĆö that *is* the "ask the user first" enforcement); the committed sample is the complete template; the same schema per `user_id` is the multi-user seam. First real code in the repo: [`scripts/search_spec.py`](../scripts/search_spec.py), [`scripts/jsearch_probe.py`](../scripts/jsearch_probe.py), [`config/search_config.sample.yml`](../config/search_config.sample.yml). ŌåÆ [ADR-0010](adr/0010-job-source-jsearch.md) (+ addendum), [02-architecture](02-architecture.md), build-plan Step 0, plan ┬¦20ŌĆō┬¦22.

## 16. Operational store ŌĆö Aurora Serverless v2 + Data API, no VPC (ADR-0014)
**Context.** D-v0-1, the one flagged Postgres sub-decision: Aurora Serverless v2 + RDS Data API **vs** RDS `t4g.micro` + VPC.
**Discussion ŌĆö the deciding insight.** The fetch Lambda *must* call the **public JSearch internet API**. A VPC-bound Lambda (required for RDS, which has no Data API) then needs a **~$32/mo NAT gateway** for that outbound call ŌĆö which erases the cheaper-DB savings and adds VPC / endpoint / proxy complexity. Aurora SLv2 + the **Data API** lets the Lambda reach the DB over **HTTPS while staying outside any VPC** ŌĆö direct access to S3 / Secrets / Bedrock / JSearch, zero networking plumbing, scales toward ~$0 idle.
**Decision.** **Aurora Serverless v2 + RDS Data API, Lambda outside any VPC.** pgvector lives in the *same* Aurora (no separate vector DB ŌĆö no OpenSearch/Pinecone). ŌåÆ [ADR-0014](adr/0014-operational-store-aurora-serverless-data-api.md), plan ┬¦26.

## 17. The silver-layer evolution ŌĆö from a cheap library to LLM dissection on everything (ADR-0015/0016)
**Context.** A deep discussion on *how* the JD body gets dissected ŌĆö the most consequential pivot of the build phase, and the clearest example of the thinking **evolving through several positions in one session.** Recorded in full because the *path* is the point.
**The trail (the "changed our minds" part).**
1. I first proposed a **`lingua` library** for the silver *language-detect* step ŌĆö and argued it well when Tarig asked for the accuracy/cost/speed comparison: English-vs-Arabic is ~99.9% separable (disjoint scripts), free, deterministic, quota-independent, vs an LLM's cost/latency/non-determinism for a *trivial* classification. For that narrow task, the library genuinely wins. We even captured it (silver = pure-Python, quota-independent ŌĆö plan ┬¦26, commit `62d7fbe`).
2. **Tarig's correction (pivotal):** we'd been talking past each other about *different stages*. He didn't mean language detection ŌĆö he meant a **full LLM dissection at the silver/landing layer**, because the structured output (skills + requirement levels, sector, normalized title, location, ŌĆ”) **populates the market-wide dimensional tables** (Skill-Demand, Sector Intel). Those analytics need **every** posting dissected, not just the gold (candidate-matched) subset ŌĆö *extracting only at gold would bias "demand" toward his own profile.* Language is merely *one byproduct field* of that dissection; `lingua` was answering a question he wasn't asking.
3. This **reverses "only gold reaches the LLM."** We weighed the cost honestly (the LLM now runs on *all* deduped postings) and chose the mitigation: **per-task models** ŌĆö a *cheap/fast* model for the bulk silver dissection, the *strong* model for scoring. Offered a near-free English pre-gate to skip non-English JDs; Tarig chose to **dissect every posting (no pre-gate)** for maximal market signal.
4. **Variation across JDs** ("required" vs "essential" vs "must-have"; "Postgres" vs "PostgreSQL"; different languages/companies) is countered by **LLM-understanding of equivalence + a canonicalization layer (`dim_skill` synonyms) + a structured-output contract (Pydantic, temp 0) + replay** over immutable bronze ŌĆö *not* by enumerating phrasings.
5. **The governing tenet that fell out** ŌĆö Tarig: *"any part of the architecture must be replaceable and upgradable **by type**, not just scaled in volume."* So every stage is a **config-selected strategy behind a port** (`SourceAdapter` ┬Ę `Dissector` ┬Ę `FilterStrategy` ┬Ę `Embedder` ┬Ę `Scorer`), generalizing the model-agnostic LLM into a pervasive principle (**P3**).
**Decisions.** Silver = **LLM `Dissector` on every deduped posting** ([ADR-0016](adr/0016-llm-dissection-at-silver.md)); gold = **LLM `FilterStrategy`** (on the dissected fields); **type-replaceability is a first-class tenet** ([ADR-0015](adr/0015-type-replaceable-pipeline-stages.md), P3). **Quota consequence (reconciled honestly):** with the LLM at silver, only **bronze** (fetch + land) is *live-runnable* until ERR-001 lifts ŌĆö silver-onward needs Bedrock ŌĆö but the whole pipeline is **build + test-able now with Bedrock mocked.** *This openly supersedes the ┬¦26 capture from earlier the same session; the un-winding of the "silver = pure-Python / quota-independent" claims is recorded, not hidden.* ŌåÆ plan ┬¦27.

## 18. ŌŁÉ Milestone ŌĆö the LLM goes live: routing around Bedrock to DeepSeek (ADR-0017)
**The moment the project stopped being blocked and started working** (2026-06-24). For weeks the entire scoring/dissection path was dark behind the Bedrock new-account quota (┬¦13, [ERR-001](ledgers/errors.md)). This section is how we broke out ŌĆö and *why the way we broke out matters more than the fix itself.*

**The decision: stop waiting, route around it.** The quota was `0` / non-adjustable, account-maturity-gated, with no timeline ŌĆö re-tested 2026-06-17 and 06-23, still throttling; an AWS Support case open with no ETA. Continuing to wait was not a plan. The architecture's whole premise (P2; [ADR-0015](adr/0015-type-replaceable-pipeline-stages.md)) is that a blocked stage is broken by the *minimal migration*, and [ADR-0012](adr/0012-model-agnostic-llm.md) had already made the LLM model-agnostic. So the move was, in hindsight, the one the design was built for: **swap the LLM transport behind the `LlmClient` port.** Tarig framed it as an engineering challenge ŌĆö *"Bedrock genuinely doesn't work, so process the data with another LLM."*

**The options, weighed honestly** (grounded in real hardware + current facts, not assumptions):
- **Local LLM (Ollama).** We detected the machine first: an **RTX 3050 Laptop GPU, 4 GB VRAM** ŌåÆ realistically 3ŌĆō4B models (fine for extraction, weak for nuanced scoring), and a laptop *can't run the deployed serverless pipeline*. $0 + fully private, but limited. A sharp catch: **"DeepSeek" can't even run locally** ŌĆö the real V3/V4 models are 671B; "local" would mean a *different* small model (Qwen/Phi), not DeepSeek.
- **DeepSeek API.** Real DeepSeek, OpenAI-compatible, dirt cheap (V4 Flash $0.14/$0.28 per M), serverless-ready. Privacy cost: China-hosted; ToS permits training on API inputs.
- **Anthropic API direct (Claude).** When Tarig asked *"why can't I just use an Anthropic API key?"* we worked it through: a Claude.ai **subscription ŌēĀ API access** (the API is separate pay-per-token billing), but a first-party Anthropic key **does** bypass Bedrock entirely ŌĆö US-hosted, **doesn't train on API data** (best privacy) ŌĆö at ~10├Ś DeepSeek's cost. Kept as the privacy/quality fallback.

**The design insight that made the choice low-stakes.** Don't build a "DeepSeek adapter" ŌĆö build **one `OpenAICompatLlmClient`**, because OpenAI-compatible is the de-facto standard (DeepSeek, Ollama, OpenRouter, vLLM all speak it). The *backend* becomes pure config (`base_url` + `api_key` + `model`). That makes the provider a one-line swap and **permanently kills the single-provider fragility Bedrock had just inflicted** ŌĆö and it's *more* minimal than the Bedrock-Converse transport we'd specced. **Tarig chose DeepSeek** for cost; the port keeps every other option a config flip away. Captured as [ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md).

**The detour ŌĆö recorded because it's exactly the kind of thing that bites the next person.** First live call: **`402 Insufficient Balance`.** DeepSeek's widely-cited "5M free signup tokens" did **not** apply to this account ŌĆö the API needs a funded balance to make *any* call. The key + integration were valid throughout (auth passed, correct endpoint + request shape); it was purely a billing gate. Tarig added a **$2 balance** and, because the key had been pasted in plaintext, **regenerated it** (rotation) and stored the fresh one in Secrets Manager himself ŌĆö the key never re-touched the chat.

**Verified live ŌĆö 2026-06-24.** `scripts/deepseek_smoke.py` (reads the key from Secrets Manager, one cheap call, never prints the key) ŌåÆ **HTTP 200 from `deepseek-v4-flash`**. The LLM path is real; the canonical model id `deepseek-v4-flash` is confirmed for config.

**What this milestone actually means** (the parts a future builder must carry forward):
- **ERR-001 is *Mitigated, not Resolved*** ŌĆö a distinction we kept honest everywhere. The Bedrock quota is *still 0*; we simply don't use Bedrock. It is a **parked, config-swappable backend**; the AWS case stays open as optional.
- **The quota scope inverted.** The ┬¦17 caveat ("only bronze is live until the quota lifts") is **gone** ŌĆö DeepSeek has no new-account gate, so the **whole pipeline (bronze ŌåÆ silver dissection ŌåÆ gold ŌåÆ score) is live-runnable.** The first real LLM code (the silver `Dissector`) is now runnable work on live data, not blocked work.
- **The architecture paid for itself.** This is type-replaceability (P3, [ADR-0015](adr/0015-type-replaceable-pipeline-stages.md)) doing exactly what it was designed to do: a blocked provider became a *config change, not a rewrite*. `BedrockŌåÆDeepSeek` is now the textbook example cited in [00-design-philosophy](00-design-philosophy.md).
- **The honest cost is on the record** ([ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md)): data now leaves AWS to a China-hosted API that may train on it. Fine for *public JD text*; the `Scorer` (which sends the CV/profile) can flip to local Ollama or Anthropic-direct via config if PII privacy ever outweighs cost.

## 20. The through-line of the build phase
Two disciplines recurred and are worth naming. **Intellectual honesty over a convenient story:** Kimi was *tested*, not assumed (it throttles via the API); "buildable now" was *narrowed* the moment the LLM moved into silver; a console "it works" was refused as proof when the API said otherwise; a decision captured at 2pm was openly reversed at 4pm when the reasoning changed; and when the LLM finally went live (┬¦18) we recorded the unglamorous detour ŌĆö DeepSeek's "free tokens" didn't apply, `402` until funded ŌĆö right alongside the win. **Match the tool to the task, then make the tool swappable:** cheap deterministic work where a library suffices, the LLM where judgment or market-wide structured extraction is the *actual* requirement ŌĆö every stage behind a port so any of these becomes a config change, not a rewrite. **┬¦18 is the proof of that second discipline:** the weeks-long Bedrock block was broken not by waiting but by a one-line provider swap the design had anticipated ŌĆö `BedrockŌåÆDeepSeek`, config not code.
