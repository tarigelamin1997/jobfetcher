# 01 · Session Decision Journal

> **Purpose:** preserve *why* JobFetcher's design is what it is. This is the narrative record of the design session that produced this repo — the reasoning, the alternatives rejected, the pivots, and the order they happened in. If you're a future session (human or agent) and you only read one doc to understand the *intent* behind the architecture, read this one. The crisp *decisions* are in [the locked-decisions ledger](ledgers/decisions-locked.md) and [`adr/`](adr/); this is the *reasoning*.

Format: each entry is **Decision · Why · Rejected · So-what.**

---

## 0. Origin & mandate

**Where we started.** A prior, very elaborate JobFetcher design existed (a 23 KB rules `CLAUDE.md`, a 75-page planning PDF, an 8-phase Notion plan, multiple standards docs, a portfolio assessment). It described a full AWS-serverless pipeline: EventBridge → Step Functions → 7 Lambdas → Bedrock scoring + CV tailoring → DynamoDB + Streams CDC → Notion (4 DBs) → SES → CloudWatch, all in Terraform, plus near-miss graduation, skill/sector analytics, a feedback hub, and a multi-user vision.

**Mandate (Tarig).** Scrap it and start fresh "in a better way." First *absorb* all existing knowledge, then *interrogate* (100–300 questions) to ground the rebuild in real intent rather than inherited assumptions, then reflect the understanding into **documentation that survives session loss**, then build.

**So-what.** This session is the interrogation + design. The deliverable is this doc set. The old artifacts were fully absorbed (their substance lives on where it earns its place) and then deleted for a clean slate.

---

## 1. The four framing forks (these reordered everything)

1. **Primary goal → "Both equally": the tool IS the portfolio.** *Why:* Tarig wants a tool he genuinely uses *and* a repo that lands a job; neither dominates. *So-what:* every component must earn *dual* value — daily-useful AND real portfolio signal. This became the knife used to cut "portfolio theater."
2. **v1 scope → "Full original system."** *Why:* he wants completeness, not a lean cut. *Rejected:* lean MVP, DE-only subset. *So-what:* discovery became *re-justify + improve* each part, not remove — later reconciled with minimalism via the evolutionary model (§9).
3. **Architecture authority → "Claude recommends (defended); Tarig approves."** *So-what:* I bring defended recommendations, not menus.
4. **Working style → "Co-design now, then I build."** *So-what:* the whole design was settled interactively before any code.

Discovery method chosen: **section-by-section interactive** (≈190 questions organized A–V).

---

## 2. Goals & what "better" means (Section A)

- **Nothing failed — "it was fine, want it better."** The restart is a smarter second pass, not a rescue. **Truly greenfield** (old ideas as inspiration only).
- **"Better" = (1) more DE depth** (SQL/warehouse/dbt — the assessment's flagged gap) **+ (2) more reliable.** These are the two north stars.
- **"Keep it all"** — Tarig initially saw nothing as overkill; complexity acceptable *if justified*. *So-what:* the burden landed on me to justify each retained piece — which later (via the defensibility audit and evolutionary model) became the project's backbone rather than a tension.

## 3. The real job-search situation (Section B)

- **Active-but-selective**, urgency **in weeks**. *So-what:* he'll actually use this soon → it must be genuinely good, and value must arrive fast. This created the first "full system but fast" tension, resolved by **MVP-first sequencing**.
- **Biggest time-saves = discovery + filtering/scoring + CV** (the core pipeline). **Not** tracking/market-intel. *So-what — key insight:* the analytics layer is *portfolio/strategic* value, not daily time-save. So sequence the core pipeline first; analytics is the DE-depth layer behind it.
- **Auto-CV = core value.** *So-what:* CV tailoring stays first-class.

## 4. Target roles, market, profile (Section C)

- **DE / Data Platform / Data Architect only** (no spread to Analytics-Eng/Cloud/ML). **Riyadh → GCC → open to relocate; NOT remote-global** (on-site oriented). **Profile accurate as-is** (use it unchanged as scoring source of truth). **English only.** Old sensitivities still hold (site offline; "Cansa Group" naming; honesty rules).
- *So-what:* simplifies search config, scoring, and CV; no Arabic handling; no re-intake of profile.

## 5. Portfolio positioning (Section D) — a pivotal lock

- **Headline = "end-to-end system design."** *So-what:* **breadth IS the portfolio point** — which makes "keep it all" *coherent* rather than greedy: the complete working system is the story.
- **Must-demonstrate = all four:** LLM/AI, warehouse/dbt modeling, AWS-serverless/IaC, streaming/CDC. *So-what:* this justified several complex pieces (CDC is a *required* signal, not theater) **and** confirmed adding a real warehouse+dbt layer.
- **Engagement = "some will clone & run"** → must be genuinely reproducible. **Presentation = README + diagram + demo video, all required.**
- **Portfolio thesis (locked):** *a complete, reproducible, production-style system across LLM + warehouse/dbt + serverless/IaC + streaming/CDC, fully documented, clone-and-runnable.* (Note: §8 honesty audit + §9 minimalism later *right-sized* how much of this is built up-front vs. reached by migration — the thesis is the destination, not the v1.)

## 6. Scope, timeline, working process (Sections E, T, U)

- **Full-tilt (20+ h/wk).** Approach initially **"plan everything, then build"** → later refined (§9) to *"plan the foundation + current stage fully; plan each migration just-in-time."*
- **Multi-user + feedback hub = design-for, build-later** (seam-ready). *So-what:* capability designed in, implementation deferred — pragmatic scoping inside "keep it all."
- **Autonomy = "confirm major decisions only."** **Testing = full pyramid** (unit + LocalStack/moto + dbt tests + live smoke). **Docs = everything (CLAUDE.md + design doc + ADRs + README + exhaustive build plan), exhaustive detail.**
- **Docs location** — a conflict surfaced (Notion vs repo) and was **resolved: repo is canonical** for all design/plan/ADR docs; Notion only hosts the operational user-facing databases. *Why:* design docs versioned with code = part of the portfolio + the context-survival requirement.

## 7. Pipeline internals (Sections H–K, M, N–S)

- **Data:** multi-source via **official APIs only** (JSearch + Adzuna to start), no scraping → a **pluggable source-adapter** layer. Volume **modest (10–30/day)** → Bedrock cost trivial; dedup is about *quality* not scale.
- **Scoring:** keep the **7-factor ATS framework** (tune weights), **explainability is critical** (strengths/gaps/strategic assessment), a **lightweight calibration loop** (capture score corrections → tune prompt + a scoring-accuracy SLO), keep the scam-gate + poster-type label.
  - **Threshold (amended post-design):** initially set at 75 with an implicit two-gate idea; **changed to a *single* user-configurable `threshold` (default 60), read at runtime, gating *both* the shortlist and CV writing** (floor 50, near-miss 10). *Why:* Tarig wants the CV-writing bar to be one knob each user tunes frequently without a redeploy, and a single gate is simpler/more defensible than two. The active threshold is stamped per run for measurement. *Rejected:* two separate thresholds (shortlist vs CV) — unnecessary complexity for one user. (See plan §12.)
- **CV:** **drop LibreOffice-in-Lambda** (the old #1 reliability risk) → a deterministic, dependency-light renderer (one content model → DOCX via python-docx + PDF via a pure-Python/HTML path). Keep the *tarig-cv* template as the refined base. **One master CV.** **Strict honesty + human-review gate** (CV is a draft you approve; the review gate doubles as calibration capture).
- **UI:** **email + Notion both** first-class; status tracking in Notion; a BI dashboard over the marts is *designed-for, build-later*.
- **Near-miss/graduation:** keep the full watch→re-score→graduate loop. **Observability:** initially "full suite," later **right-sized** (§8). **Orchestration:** Step Functions (AWS-native) — but it became a *migration* (M3), not a v0 component (§9).

## 8. The architecture synthesis, then four deep pivots

I synthesized a **two-plane architecture** — an **operational plane** (the daily serverless tool) cleanly separated from an **analytical plane** (DE-depth: warehouse + dbt). *Why:* lets us push hard on the DE-depth prime directive without diluting the serverless story; realizes the medallion as a *real dbt-modeled warehouse*, not just S3 folders. Tarig approved it "with tweaks," then drove four deep pivots:

### 8a. Warehouse: Snowflake vs Databricks vs Postgres
*Decision:* **Snowflake** as the (eventual, conditional) warehouse — **not** Databricks. *Why:* the assessment's flagged gap is **SQL/warehouse modeling + dbt**, which is Snowflake's lane; at 10–30 rows/day the warehouse is a *portfolio/skill* choice, not a data need, and Snowflake is the cheapest/most-reliable/cleanest-with-dbt option, with strong KSA/GCC demand. *Rejected:* **Databricks** — Spark-on-tiny-data is unconvincing and a weaker fit; the Spark/Delta signal belongs in the sibling **OrderFlow** project where volume justifies it. *Crucial reframe:* since we already run **Postgres**, the entire medallion could live there for $0 — so the warehouse choice is purely about signal, which later (§9 tool-minimalism) made **Snowflake conditional** and **Postgres+dbt the default**. See [ADR-0004](adr/0004-warehouse-strategy.md).

### 8b. Operational store: Postgres + (deferred) Debezium CDC, replacing DynamoDB+Streams
*Decision:* **managed PostgreSQL** as the operational store. *Why:* the data is inherently relational (clusters → postings → scores → applications); a relational DB is the *right-tool fit*, and it mirrors Tarig's real DE expertise. *Rejected:* DynamoDB + Streams — NoSQL for relational data was the *weaker* original choice. **Debezium CDC** was adopted then **deferred**: at this volume, nightly batch → warehouse + incremental dbt is right-sized; Debezium becomes the *documented scale-up path* ("we'll need it later"), and the built CDC showcase lives in OrderFlow. See [ADR-0003](adr/0003-postgres-over-dynamodb.md), [ADR-0009](adr/0009-batch-not-debezium-v0.md).

### 8c. Deduplication: cluster-and-surface, never hide
*The 99.99%-accuracy question.* Tarig wanted near-perfect dedup across multiple APIs. *Reframe (honest):* a literal 99.99% guarantee on fuzzy job-text matching isn't honest. There are two error types — a **missed duplicate** (trivial cost: one extra cheap score) and a **wrong merge** (the dangerous one: a real job gets hidden). *Decision (Tarig's, and it overrides the old "pick one canonical + hide duplicates"):* **never silently hide a posting.** Instead **cluster** suspected-same postings and **surface the whole group** with every platform's apply-link + a "suspected same as X/Y/Z" note; the **user decides** whether to apply via one or several (hiring teams sometimes favor one platform's pipeline). Uncertain clusters go to a dedicated **Suspected-Duplicates** surface to confirm/split. *Engineering:* precision-first + fail-safe + **measured precision/recall** (entity resolution with real numbers — a strong DE signal); JD-body embeddings (pgvector) + apply-URL match + company canonicalization + time-window scoping; **score + tailor a CV once per cluster** (identical content) but show all links. See [ADR-0005](adr/0005-dedup-cluster-and-surface.md).

### 8d. Distribution: self-hosted / open-source (not SaaS)
*Decision:* **self-hosted / open-source.** *Why:* the goal is a job-search tool + a portfolio, **not a company.** Self-hosting makes the IaC + reproducibility *itself* the portfolio value, with zero ongoing liability. *Rejected:* **centralized SaaS** — it would mean owning others' cost, auth, billing, support, multi-tenancy, and **other people's CVs/PII** (real legal/privacy liability) plus job-data ToS exposure: a startup, not a portfolio piece, and a distraction from landing a job. Multi-user stays a documented future pivot. See [ADR-0007](adr/0007-self-hosted-distribution.md).

## 9. The honesty audit, the evolutionary reframe, and the governing principles

**Honesty audit (Tarig's challenge: "can you defend this architecture, or are we spinning service names?").** I audited every component against a **defensibility rubric** (now in [00-design-philosophy](00-design-philosophy.md)). Outcome: most pieces are defensible as right-tool-fit; **two were flagged** — Debezium-CDC (right-sized to batch, deferred) and the full SLO/alarm suite (**right-sized** to a few real alarms + documented SLOs). Adopted dial: **BALANCED** — simplest defensible option by default, 2–3 labeled showcases (warehouse/dbt + measured entity-resolution).

**Evolutionary architecture reframe (Tarig — the single biggest idea).** Don't build the full system at once. Build the **minimal working core (v0)**, then evolve through a **sequence of deliberate, observable migrations — each a clean GitHub release.** *Why it's the strongest framing:* controlled architectural *evolution* (an ADR + migration guide per step) is a rare senior/staff portfolio signal; it makes every piece defensible *by construction* (capability arrives exactly when justified); and a leaner v0 yields *more, better* migrations — the migrations ARE the showcase. *So-what:* "full system" becomes the *destination reached via migrations*, fully reconciling "keep it all" with "absolute minimalism."

**The roadmap is directional, not fixed (Tarig).** You can't draw the full sequence before shipping — each stage's *implementation* is the bottleneck that reveals the next capability. So we commit firmly to only: v0, the migratable architecture, and release discipline. The rest is a living hypothesis. This refined "plan everything then build" → **"plan the foundation + current stage fully; plan each migration just-in-time."**

**Governing principles P1/P2 (Tarig).** **P1 absolute minimalism** (minimal complexity for the present problem; complexity is entropic → resist it) + **P2 bottleneck-driven evolution** (solve the top bottleneck with the minimal migration; the migration-decision protocol). Together: *add the minimum, only to break the biggest real bottleneck.*

**Tool-minimalism wins (Tarig).** Only build what a real **tool** bottleneck justifies; DE-depth is the *tiebreaker*, not a license to add. *So-what — this revised an earlier "lock":* **Snowflake became conditional**, with **Postgres + dbt as the default** analytics; Step Functions survives only because it's *earned* by real Lambda complexity (M3). DE-depth is still fully served, minimally and honestly (Postgres + dbt modeling/tests/lineage + measured entity resolution + data contracts + the evolutionary-architecture story itself).

## 10. v0 boundary & the roadmap

- **v0 (locked) = the irreducible core:** EventBridge (daily) → **one Lambda**: fetch from **one source** → raw to **S3** + rows to **Postgres** → **Bedrock** score vs profile → **daily email digest**. Terraform + Secrets Manager + tests + minimal CI. *No CV, no Notion, no Step Functions, no dedup-clustering, no warehouse.* It works (you wake to a scored shortlist); everything else is a migration.
- **CV = M1** (first migration), **email-only** for v0, **single Lambda** for v0 — chosen *leanest*, because a leaner v0 creates more (better) migration stories.
- **Roadmap (directional):** v0 → M1 CV → M2 multi-source+clustering-dedup → M3 Step Functions → M4 Notion+near-miss → M5 dbt marts on Postgres → M6 skill/sector intel → M7 observability+calibration → M8 CI/CD+README/diagram/demo (= v1.0.0). Future: Debezium CDC, multi-user, feedback hub, BI dashboard, Snowflake (conditional). **Semver:** v0.x per migration → v1.0.0 at M8. Full detail + the bottleneck protocol in [03-roadmap](03-roadmap.md).

## 11. Methodology & clean slate

- **Methodology adopted (right-sized)** from Tarig's *Master Project Implementation Plan* + *Modern DE Best Practices*: documentation-as-infrastructure, What/Why/So-what, ADRs-with-rejected-alternatives, the Five-Questions error log, ledgers (phase index, contracts, locked decisions), behavioral validation gates (positive + negative), safety-first. Right-sized for solo/minimalist scale; **enforcement machinery (slash-commands vs Makefile vs checklists) is emergent** — evaluated during implementation. Full breakdown in [05-methodology](05-methodology.md).
- **Clean slate (Tarig, explicit):** delete the entire old project tree and start with nothing but this doc set. Done. The plan file and memory live outside the project and survived.

## Appendix — the through-line

The session repeatedly converged on one idea from multiple directions: **earn complexity, don't assume it.** "Both equally" → every piece earns dual value. "Defensibility rubric" → every piece beats its simpler alternative. "Evolutionary architecture" → every piece arrives only when a bottleneck justifies it. "Tool-minimalism wins" → the tool's needs gate what's built. These are four expressions of the same discipline, now encoded in [00-design-philosophy](00-design-philosophy.md).

## 12. AWS identity — from keyless root to a non-root dev user
**Context.** Setting up local AWS access surfaced that the working profile (`samareltayeb`, the account **root**) had **no static access keys** — it authenticates via a **session login** (temporary STS creds — access key + secret + *session token* — cached under `~/.aws/login/cache/`, refreshed at sign-in). Tarig, reasonably, read "no key/secret" as "no auth" and thought we had to create keys. That became a teaching thread: AWS auth is *temporary credentials*, not necessarily stored secrets.
**Discussion / what we weighed.** Tarig wanted access keys. The hard line: **never create *root* access keys** — a leak = total, unrecoverable account takeover (AWS's #1 anti-pattern). The session-login worked for the CLI, but the **AWS Toolkit extension** kept dropping (its connection expired separately). Options: (a) keep keyless root session login; (b) create a **non-root IAM user** with its own access key. We chose (b) — it is the access-key method Tarig wanted, done right, *and* AWS's own "stop using root" best practice. (An interim step first made keyless `samareltayeb` the persistent default via user env vars — plan §18 — before the IAM user replaced it.)
**Decision.** Created the non-root IAM admin user **`jobfetcher-dev`** with an access key; made it the **default for all local dev** (`AWS_PROFILE=jobfetcher` persistent + `[default]` mirrors the same key, so every resolution path — flags, env, new shell — lands on `…:user/jobfetcher-dev`); root retained only for rare root-only ops via an explicit `--profile samareltayeb`. Codified as a durable standard in CLAUDE.md: **humans → `jobfetcher-dev`; machines (CI / Lambda runtime) → their own least-privilege IAM roles, never the personal key.** All keys stay out of the repo and out of any cloud routine. Verified post-VS-Code-restart (Toolkit status bar showed `profile:jobfetcher`). → plan §18–§19, [decisions-locked](ledgers/decisions-locked.md) (Security), CLAUDE.md. *(The two-tier auth model — local session/IAM-user temp creds vs Lambda execution roles in production — is documented in the build-plan Prerequisites; no static key exists anywhere in the deployed system.)*

## 13. The Bedrock quota wall (ERR-001) and the model-agnostic response
**Context.** "Can't use Anthropic models." A read-only investigation (account `198592435375`, us-east-1) found *two* distinct failures: **(a)** the base model ids (`anthropic.claude-…`) reject on-demand → the Claude 4.x models are **inference-profile-only** (the `us.anthropic.…` cross-region id works); **(b)** — the real wall — the account's **per-day Bedrock token quota is `0` and `Adjustable = False` for *every* model.** Ruled out billing (valid card + credits) and IAM (root *and* the admin user both hit it). It is a **new-account maturity gate**, independent of billing.
**Discussion.** Tarig: "Kimi K2 Thinking works fine — use it instead of Anthropic, and make switching models never a bottleneck." So we locked **model-agnostic LLM** ([ADR-0012](adr/0012-model-agnostic-llm.md)): one `LlmClient` port over Bedrock **Converse**, model id in config **per task** (`scoring_model`, `extraction_model`, …), structured output via prompt + Pydantic (portable, not provider-specific JSON modes). **But** verification cut against the easy story: `moonshot.kimi-k2-thinking` is **ACTIVE + ON_DEMAND** (no inference-profile gotcha — a real simplicity win) yet its daily-token quota *also* reads `0` — the wall is **account-wide, not Anthropic-specific.** A 1-token `converse` test confirmed it: **Kimi throttles via the `bedrock-runtime` API exactly like Anthropic.** Where Kimi "worked" for Tarig was the **console playground** (separate limits), not the API the pipeline uses. Re-tested **2026-06-23** (≈1 week on) — still `ThrottlingException`; we refused to mark it resolved on a console anecdote.
**Decision.** **Kimi K2 Thinking is the chosen model** (config-swappable; no longer waiting on Anthropic) — but choosing Kimi settles the *model*, not the *quota*. ERR-001 remains the one blocker on the scoring path. Filed **AWS Support case `178220019100382`** to raise the new-account quota (the only deterministic lever — quotas are non-adjustable via the console); no fixed lift timeline; we **re-test in-session** each resume (a cloud cron was declined — it would require putting the personal AWS key in a cloud env, against the security model). → [ERR-001](ledgers/errors.md), [ADR-0012](adr/0012-model-agnostic-llm.md), plan §16–§17, §24–§25. **The wall held for weeks — §18 is how we got off it (we stopped waiting and routed around it to DeepSeek).**

## 14. Ingestion — query strategy, the JSearch reality, and the user-input contract
**Context.** The first concrete ingestion work: how to query JSearch within a rationed quota, proven on real data — and how the user specifies *what* to search. The principle that fell out: **free tier = learn the strategy; Pro = run it.**
**Discussion / findings (all from the live API).**
- **Query matrix:** 3 core titles (`Data Engineer`, `Data Platform Engineer`, `Data Architect`) × 6 GCC countries (`sa, ae, qa, kw, bh, om`) = **18 base queries**; `date_posted=month` as a one-time **backfill** window (explicitly distinct from a future daily-incremental window). Free Basic = 200 req/mo; one all-GCC sweep ≈ 40–70 req → **fits free** (this sweep *is* the probe); Pro (10k/mo, $25) deferred until the probe gives evidence. "Backfill only for now."
- **`language=en` is mandatory:** without it, `country=sa` returns metadata in **Arabic** *and* leaves `job_city`/`job_country` + the UTC timestamp **null**; with it they populate cleanly. A real probe call confirmed full JD text arrives (≈1,230 chars) — the critical "not truncated" check that disqualifies a source.
- **Location model (honest):** the **`country` query param is the authoritative geo scope** (per-record `job_country` is unreliable / often null); `job_location` is a denormalized city string; **there is no employer-HQ location** — JSearch gives only the job's location. City/state are therefore **gold filters**, not query params (pull broadly by country → filter cities in gold → every city still lands in bronze for analytics).
- **Secrets:** adopt **AWS Secrets Manager** for all secrets — convention `jobfetcher/<service>` (e.g. `jobfetcher/jsearch`), JSON value, read by scripts *and* Lambdas; one secret serves local probe + production (store-once, use-everywhere). The key is never committed.
**Decision.** **Search targeting = a validated, fully-explicit per-user `SearchSpec`** (Pydantic; *every* field required — "nothing taken for granted as input"): `job_titles` → query text, `countries` → the `country` param, `cities`/`states` → gold filters, plus knobs (`date_posted`, `language`, `employment_types`, `remote`, `threshold`) + budget (`max_pages_per_query`, `request_budget_per_run`). Intake = a gitignored config file the user fills, **validated loudly at load** (a missing field fails — that *is* the "ask the user first" enforcement); the committed sample is the complete template; the same schema per `user_id` is the multi-user seam. First real code in the repo: [`scripts/search_spec.py`](../scripts/search_spec.py), [`scripts/jsearch_probe.py`](../scripts/jsearch_probe.py), [`config/search_config.sample.yml`](../config/search_config.sample.yml). → [ADR-0010](adr/0010-job-source-jsearch.md) (+ addendum), [02-architecture](02-architecture.md), build-plan Step 0, plan §20–§22.

## 15. Process machinery — the gate trio + branch/PR (ADR-0013)
**Context.** Crossing from design into building triggered the previously-**deferred** "enforcement is emergent" decision (§11 / 05-methodology). The trigger — *building, inside Claude Code where a command is near-free* — was now met. Reviewing Tarig's **Master Project Implementation Plan** (Notion), we pulled the delta worth adopting now.
**Discussion.** We'd already absorbed the plan's *core* (ledgers, ADRs, the behavioral-gate standard = VG1–8). The new surface was the **enforcement commands** + the human checkpoints — right-sized: the **3 gates**, not the full 10-command catalog; **self-review**, not an external-reviewer hard gate; the six-angle chaos matrix stays cut.
**Decision.** **Gate trio as committed Claude Code slash-commands** — `/start-step` (entry: prereqs closed, spec has no `[TO BE FILLED]`, every validation criterion behavioral + negative) · `/review-step` (code: lint, secret-scan, tests, every negative case *executed*) · `/close-step` (exit: docs + ADRs updated, gate run positive **and** negative, no open `ERR-NNN`, interface contracts verified). **Two human checkpoints** (A: spec approved before code = plan mode; B: approval before the irreversible merge/tag). **Branch + PR + protected `main`** for v0 code; docs may go direct for speed. Dogfooded immediately — the adoption itself shipped through **PR #1**. → [ADR-0013](adr/0013-enforcement-gate-trio-branch-pr.md), [05-methodology](05-methodology.md), `.claude/commands/`, plan §23.

## 16. Operational store — Aurora Serverless v2 + Data API, no VPC (ADR-0014)
**Context.** D-v0-1, the one flagged Postgres sub-decision: Aurora Serverless v2 + RDS Data API **vs** RDS `t4g.micro` + VPC.
**Discussion — the deciding insight.** The fetch Lambda *must* call the **public JSearch internet API**. A VPC-bound Lambda (required for RDS, which has no Data API) then needs a **~$32/mo NAT gateway** for that outbound call — which erases the cheaper-DB savings and adds VPC / endpoint / proxy complexity. Aurora SLv2 + the **Data API** lets the Lambda reach the DB over **HTTPS while staying outside any VPC** — direct access to S3 / Secrets / Bedrock / JSearch, zero networking plumbing, scales toward ~$0 idle.
**Decision.** **Aurora Serverless v2 + RDS Data API, Lambda outside any VPC.** pgvector lives in the *same* Aurora (no separate vector DB — no OpenSearch/Pinecone). → [ADR-0014](adr/0014-operational-store-aurora-serverless-data-api.md), plan §26.

## 17. The silver-layer evolution — from a cheap library to LLM dissection on everything (ADR-0015/0016)
**Context.** A deep discussion on *how* the JD body gets dissected — the most consequential pivot of the build phase, and the clearest example of the thinking **evolving through several positions in one session.** Recorded in full because the *path* is the point.
**The trail (the "changed our minds" part).**
1. I first proposed a **`lingua` library** for the silver *language-detect* step — and argued it well when Tarig asked for the accuracy/cost/speed comparison: English-vs-Arabic is ~99.9% separable (disjoint scripts), free, deterministic, quota-independent, vs an LLM's cost/latency/non-determinism for a *trivial* classification. For that narrow task, the library genuinely wins. We even captured it (silver = pure-Python, quota-independent — plan §26, commit `62d7fbe`).
2. **Tarig's correction (pivotal):** we'd been talking past each other about *different stages*. He didn't mean language detection — he meant a **full LLM dissection at the silver/landing layer**, because the structured output (skills + requirement levels, sector, normalized title, location, …) **populates the market-wide dimensional tables** (Skill-Demand, Sector Intel). Those analytics need **every** posting dissected, not just the gold (candidate-matched) subset — *extracting only at gold would bias "demand" toward his own profile.* Language is merely *one byproduct field* of that dissection; `lingua` was answering a question he wasn't asking.
3. This **reverses "only gold reaches the LLM."** We weighed the cost honestly (the LLM now runs on *all* deduped postings) and chose the mitigation: **per-task models** — a *cheap/fast* model for the bulk silver dissection, the *strong* model for scoring. Offered a near-free English pre-gate to skip non-English JDs; Tarig chose to **dissect every posting (no pre-gate)** for maximal market signal.
4. **Variation across JDs** ("required" vs "essential" vs "must-have"; "Postgres" vs "PostgreSQL"; different languages/companies) is countered by **LLM-understanding of equivalence + a canonicalization layer (`dim_skill` synonyms) + a structured-output contract (Pydantic, temp 0) + replay** over immutable bronze — *not* by enumerating phrasings.
5. **The governing tenet that fell out** — Tarig: *"any part of the architecture must be replaceable and upgradable **by type**, not just scaled in volume."* So every stage is a **config-selected strategy behind a port** (`SourceAdapter` · `Dissector` · `FilterStrategy` · `Embedder` · `Scorer`), generalizing the model-agnostic LLM into a pervasive principle (**P3**).
**Decisions.** Silver = **LLM `Dissector` on every deduped posting** ([ADR-0016](adr/0016-llm-dissection-at-silver.md)); gold = **LLM `FilterStrategy`** (on the dissected fields); **type-replaceability is a first-class tenet** ([ADR-0015](adr/0015-type-replaceable-pipeline-stages.md), P3). **Quota consequence (reconciled honestly):** with the LLM at silver, only **bronze** (fetch + land) is *live-runnable* until ERR-001 lifts — silver-onward needs Bedrock — but the whole pipeline is **build + test-able now with Bedrock mocked.** *This openly supersedes the §26 capture from earlier the same session; the un-winding of the "silver = pure-Python / quota-independent" claims is recorded, not hidden.* → plan §27.

## 18. ⭐ Milestone — the LLM goes live: routing around Bedrock to DeepSeek (ADR-0017)
**The moment the project stopped being blocked and started working** (2026-06-24). For weeks the entire scoring/dissection path was dark behind the Bedrock new-account quota (§13, [ERR-001](ledgers/errors.md)). This section is how we broke out — and *why the way we broke out matters more than the fix itself.*

**The decision: stop waiting, route around it.** The quota was `0` / non-adjustable, account-maturity-gated, with no timeline — re-tested 2026-06-17 and 06-23, still throttling; an AWS Support case open with no ETA. Continuing to wait was not a plan. The architecture's whole premise (P2; [ADR-0015](adr/0015-type-replaceable-pipeline-stages.md)) is that a blocked stage is broken by the *minimal migration*, and [ADR-0012](adr/0012-model-agnostic-llm.md) had already made the LLM model-agnostic. So the move was, in hindsight, the one the design was built for: **swap the LLM transport behind the `LlmClient` port.** Tarig framed it as an engineering challenge — *"Bedrock genuinely doesn't work, so process the data with another LLM."*

**The options, weighed honestly** (grounded in real hardware + current facts, not assumptions):
- **Local LLM (Ollama).** We detected the machine first: an **RTX 3050 Laptop GPU, 4 GB VRAM** → realistically 3–4B models (fine for extraction, weak for nuanced scoring), and a laptop *can't run the deployed serverless pipeline*. $0 + fully private, but limited. A sharp catch: **"DeepSeek" can't even run locally** — the real V3/V4 models are 671B; "local" would mean a *different* small model (Qwen/Phi), not DeepSeek.
- **DeepSeek API.** Real DeepSeek, OpenAI-compatible, dirt cheap (V4 Flash $0.14/$0.28 per M), serverless-ready. Privacy cost: China-hosted; ToS permits training on API inputs.
- **Anthropic API direct (Claude).** When Tarig asked *"why can't I just use an Anthropic API key?"* we worked it through: a Claude.ai **subscription ≠ API access** (the API is separate pay-per-token billing), but a first-party Anthropic key **does** bypass Bedrock entirely — US-hosted, **doesn't train on API data** (best privacy) — at ~10× DeepSeek's cost. Kept as the privacy/quality fallback.

**The design insight that made the choice low-stakes.** Don't build a "DeepSeek adapter" — build **one `OpenAICompatLlmClient`**, because OpenAI-compatible is the de-facto standard (DeepSeek, Ollama, OpenRouter, vLLM all speak it). The *backend* becomes pure config (`base_url` + `api_key` + `model`). That makes the provider a one-line swap and **permanently kills the single-provider fragility Bedrock had just inflicted** — and it's *more* minimal than the Bedrock-Converse transport we'd specced. **Tarig chose DeepSeek** for cost; the port keeps every other option a config flip away. Captured as [ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md).

**The detour — recorded because it's exactly the kind of thing that bites the next person.** First live call: **`402 Insufficient Balance`.** DeepSeek's widely-cited "5M free signup tokens" did **not** apply to this account — the API needs a funded balance to make *any* call. The key + integration were valid throughout (auth passed, correct endpoint + request shape); it was purely a billing gate. Tarig added a **$2 balance** and, because the key had been pasted in plaintext, **regenerated it** (rotation) and stored the fresh one in Secrets Manager himself — the key never re-touched the chat.

**Verified live — 2026-06-24.** `scripts/deepseek_smoke.py` (reads the key from Secrets Manager, one cheap call, never prints the key) → **HTTP 200 from `deepseek-v4-flash`**. The LLM path is real; the canonical model id `deepseek-v4-flash` is confirmed for config.

**What this milestone actually means** (the parts a future builder must carry forward):
- **ERR-001 is *Mitigated, not Resolved*** — a distinction we kept honest everywhere. The Bedrock quota is *still 0*; we simply don't use Bedrock. It is a **parked, config-swappable backend**; the AWS case stays open as optional.
- **The quota scope inverted.** The §17 caveat ("only bronze is live until the quota lifts") is **gone** — DeepSeek has no new-account gate, so the **whole pipeline (bronze → silver dissection → gold → score) is live-runnable.** The first real LLM code (the silver `Dissector`) is now runnable work on live data, not blocked work.
- **The architecture paid for itself.** This is type-replaceability (P3, [ADR-0015](adr/0015-type-replaceable-pipeline-stages.md)) doing exactly what it was designed to do: a blocked provider became a *config change, not a rewrite*. `Bedrock→DeepSeek` is now the textbook example cited in [00-design-philosophy](00-design-philosophy.md).
- **The honest cost is on the record** ([ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md)): data now leaves AWS to a China-hosted API that may train on it. Fine for *public JD text*; the `Scorer` (which sends the CV/profile) can flip to local Ollama or Anthropic-direct via config if PII privacy ever outweighs cost.

## 21. The agentic build pipeline in practice — and the independent-verifier stage that earned its place
**Context.** §15 committed the gate *trio* (start → review → close); [ADR-0019](adr/0019-agentic-build-orchestration.md) had since framed the build as an **agentic per-unit pipeline** — Builder → Reviewer → Scribe → Guardian, fanned out across independent units. Part 3 is where that pipeline ran *for real*, across C-2 (storage), C-3 (infra), Step 4 (fetch→silver), 4b (gold), Step 5 (scorer), Step 6 (notifier). This section is the one process learning the build itself forced.

**The bias insight (Tarig's, and it is the pivotal one).** An in-build Reviewer that the *orchestrator* spins up, prompts, and judges **shares the orchestrator's blind spots** — it is framed by the same context that produced the code, so it tends to confirm rather than break. A review you commission to grade your own work is not independent in the way that matters. *So-what:* a Reviewer inside the loop can report "looks good" precisely *because* it was set up to look at what the builder looked at.

**The response — a fresh-context adversarial Independent Verifier (ADR-0019 amendment).** Add a stage that is **not pre-framed**: a verifier given the spec and the diff cold, with one instruction — *try to break this* — and no exposure to the builder's or reviewer's reasoning. It is adversarial by mandate, independent by construction.

**It was validated by asymmetry — the evidence the stage has teeth.** The proof isn't that the verifier always finds something; it's that it finds things *when they are there* and clears the code *when they aren't*:
- **Step 4 (fetch→silver):** the in-build Reviewer reported **"0 must-fixes."** The fresh verifier — same diff, cold context — found **3 crash blockers**: a malformed-shape response, a non-JSON response, and a read-timeout all violated the **"never crash the run"** contract (the pipeline must degrade, not die, on a bad upstream reply). CodeRabbit found further items. The in-loop Reviewer had missed all three because it was reasoning along the builder's happy path.
- **Gold (4b), Scorer (5), Notifier (6):** the *same* verifier, genuinely trying to break the code, confirmed **clean** — and on the notifier caught a real **should-fix** (the apply-URL `href` needed a scheme allowlist). A verifier that found 3 blockers on Step 4 and then honestly passed three other units is a verifier that is *actually testing*, not rubber-stamping in either direction.

**So-what.** Three independent eyes per unit — the **fresh verifier + CodeRabbit + the human** — none of them spun by the orchestrator that wrote the code. The cost is one extra agent stage per unit; the payoff is the Step-4 crash-blockers caught *before* merge. This is logged as a process improvement, not just a code outcome: **independent verification beats self-assessment**, and the build phase produced the receipt.

## 22. Storage and infra, validated against reality (C-2, C-3)
**Context.** Before any pipeline stage could persist or run, v0 needed a schema + a storage port (C-2) and the cloud infra to host them (C-3). The discipline both units shared: **validate against the real thing, don't assume from the design.**

**C-2 — schema + `Repository`, live-validated on real Postgres.** The v0 schema (Alembic migrations) + a `Repository` port implemented over **SQLAlchemy Core with the aurora-data-api dialect** (so local code speaks the same Data API the Lambda will use in production). The deciding test choice: **validate on a dedicated local Postgres** — a `jobfetcher-db` docker-compose service — **not LocalStack**, because LocalStack does not meaningfully mock the RDS Data API, so a green LocalStack run would prove nothing about the path that actually ships. The DB round-trip was proven **5/5 on the real Postgres**, which is what let us *revoke* the deviation that had been carrying C-2 on faith.

**C-3 — Terraform infra, apply-validated end-to-end then destroyed to ~$0.** The infra from [ADR-0014](adr/0014-operational-store-aurora-serverless-data-api.md) (Aurora Serverless v2, scale-to-zero, Data API, **no VPC**; least-privilege IAM) was **actually applied** — Aurora came up on **PG 16.6**, the **Data API answered over HTTPS** with the Lambda outside any VPC, and the IAM role was confirmed to carry **no Bedrock permissions** (the §18 swap to DeepSeek means Bedrock is a parked backend, so the role doesn't grant it). Then the whole stack was **`terraform destroy`-ed back to ~$0**. *Why apply-then-destroy rather than plan-only:* the cost story ("Aurora scales toward $0 idle") and the no-NAT-gateway claim are only *proven* by standing the real thing up and tearing it down — a `terraform plan` proves syntax, not behavior or bill. We bought the proof and then stopped paying for it.

**So-what.** Both foundation units rest on a *behavioral* check against the production-shaped target (real Data API dialect on real Postgres; real Aurora apply over real HTTPS), consistent with the project's "a liveness check is no gate" rule (§15).

## 23. Ingestion to the gold layer — and a defended deviation from the build plan (Step 4, 4b)
**Context.** Step 4 productionized the ingestion path; Step 4b built the gold layer. Step 4b is where the build plan said one thing and we did another — *on purpose, and on the record.*

**Step 4 — fetch → bronze → silver.** This wired the validated probe (§14) and the LLM `Dissector` (§17) into the real pipeline: fetch from JSearch → land immutable bronze → dissect to silver. The crash-blockers the independent verifier caught here (§21) were exactly the upstream-reliability cases this stage has to survive — bad shapes, non-JSON, timeouts — and they were fixed before the unit closed.

**Step 4b — the `Profile` contract + the gold `FilterStrategy`, and the deterministic-default deviation.** Two decisions:
1. **v0 gold filter = deterministic, not LLM — a defended deviation from the build plan.** The plan ([ADR-0016](adr/0016-llm-dissection-at-silver.md)/§17 era) specced an **LLM `FilterStrategy`** at gold. But at **10–30 jobs/day**, an LLM gold-filter is **redundant with the Scorer** — the Scorer is already an LLM pass over each candidate, so an LLM gate *in front of it* spends a second LLM call to pre-judge what the next LLM call judges anyway. **P1 (absolute minimalism)** says cut it. *So we made the v0 default a deterministic filter* (cheap, quota-free, deterministic) — **and still built the LLM strategy**, config-selectable behind the same `FilterStrategy` port (§17's P3). This is a **deviation from the written plan, defended not hidden:** the plan's intent (a gold filter that selects candidates) is honored; the *implementation* is right-sized to the volume, and the planned LLM version is one config flip away the moment volume or filtering nuance justifies it.
2. **1:1 clusters (`cluster_id = posting_id`) to close a real gap.** The score and downstream tables key on `cluster_id`, but **v0 has no clustering** (clustering is M2 — §10, Diagram 2). Rather than leave `cluster_id` dangling or invent a fake clustering, v0 sets **`cluster_id = posting_id`** — every posting is its own singleton cluster. This makes the `cluster_id` contract *real and satisfiable today* while keeping the exact seam M2's real clustering will grow into (one posting → one cluster becomes many postings → one cluster). The schema doesn't change at M2; only the population of `cluster_id` does.

**So-what.** Both choices are the same move: **honor the contract, right-size the implementation to v0's volume, keep the upgrade seam.** The deterministic filter and the singleton cluster are not shortcuts — they are the minimal correct v0 of a seam designed to grow.

## 24. The Scorer — and the VG3 honesty call (Step 5)
**Context.** Step 5 built the heart of the tool: the LLM `Scorer` that turns a candidate posting + the profile into a 0–100 fit score. It is also where a validation gate had to be **openly softened** rather than faked — the cleanest example in the build phase of intellectual honesty over a convenient green check.

**What was built.** A **7-factor ATS** scorer, **temperature 0**, on **`deepseek-v4-pro`** (the strong model — cheap-fast is reserved for the bulk silver dissection per §17). Plus two correctness guarantees: **migration 0002 added `UNIQUE(score.cluster_id)`**, making the 1:1 invariant from §23 a *database* guarantee (not just application convention); and **VG8 (threshold-is-config)** was proven **behaviorally** — change the config threshold, watch the shortlist boundary move, no redeploy.

**The honest finding (VG3).** VG3 was specced as a *determinism* gate — same input → same score. The build found that **`deepseek-v4-pro` is non-deterministic even at temperature 0**: on **identical input** we observed score deltas of **0, 6, 10, 13, 14, and 27**. The cause is the model, not the code — MoE expert-routing and floating-point non-associativity make even a temp-0 MoE model jittery across calls. **The independent verifier confirmed the code is correct** — temperature 0 *is* actually sent in the request; we are not accidentally sampling.

**The decision — VG3 → best-effort, stated plainly.** Since the *code* is right and the *model* is the source of jitter, we did **not** fake determinism. We **softened VG3 to best-effort**: the guaranteed invariant is "**temp 0 is sent**"; the v0 score is positioned as a **triage signal, not a precise number** — good enough to rank and threshold a day's 10–30 jobs, not claimed as a stable measurement. Precise stability + calibration is **deferred to M7** (the `score_override` accuracy loop — §6/§9), where capturing human corrections is the *right* place to pin down scoring accuracy anyway.
- **Rejected — switch the scoring model.** A more deterministic model exists, but switching deviates from [ADR-0017](adr/0017-llm-transport-openai-compatible-deepseek.md) (DeepSeek, OpenAI-compatible) for a property v0 doesn't actually need yet.
- **Rejected — average N samples per posting.** Sampling K times and averaging would reduce jitter but multiplies LLM cost K× for a triage number at 10–30 jobs/day — **anti-P1** at this volume.
- **⏳ Follow-up (pending — Step 7 branch).** The first implementation set the live test to a **~±20** sanity band, but a later run drifted **27** — the band itself is too tight, so the live determinism test can flake. The pending refinement: **drop the hard delta assertion**, keep only the temp-0-is-sent assertion, and **log** the delta as an observation — the truly honest "assert what we control, observe what we don't." Captured in [04-v0-build-plan](04-v0-build-plan.md) Step 5; not blocking (the live test runs only locally with a key; CI skips it).

**So-what.** VG3 is the build phase's proof that a validation gate is a *measurement*, not a box to tick: when the measured behavior didn't match the gate, we changed the *gate's claim* to the truth (temp-0-sent, score-is-triage) and moved the precision work to where it belongs (M7), rather than pretending determinism we don't have.

## 25. The Notifier — the daily digest, and an un-bypassable injection guard (Step 6)
**Context.** Step 6 is the last pipeline stage of v0's core: turn the threshold-gated shortlist into the email you wake up to.

**What was built.** A **SES** daily digest in **HTML + plaintext** (multipart, so any mail client renders it) of the shortlist — the postings at or above the configured threshold. Two correctness properties:
- **VG5 — the zero-matches case is a first-class output.** A day with **no postings above threshold** does not produce an error or a silent no-send; it sends a valid **"no matches today"** email. The tool reporting *nothing-good-today* is a feature, not a failure mode — you still know the run happened.
- **The apply-URL injection guard (the independent verifier's should-fix from §21).** `apply_url` is **untrusted input from JSearch** rendered into an HTML `href`. Left raw, a hostile or malformed URL (`javascript:`, `data:`, …) is an email-injection vector. The guard **allowlists the `href` scheme to `http`/`https` only** — anything else is rejected, not sanitized-and-hoped. The independent verifier, tasked with breaking it, **could not bypass the allowlist** — which is the bar an injection guard has to clear to count.

**So-what.** The notifier closes the v0 loop (fetch → … → score → *email in your inbox*) and does it with the same two habits as the rest of the build: the unglamorous edge case (zero matches) is handled as a real output, and the security guard on untrusted input is **proven** against an adversary, not merely present.

## 27. v0.1.0 — the deploy, and why the live run was irreplaceable (Step 10)
**Context.** Step 10 is the one that turns "code-complete" into "real": `terraform apply` the v0 stack onto live AWS, create the schema on Aurora, run the whole pipeline against the real internet, watch the email land — then `terraform destroy` back to ~$0. It is also the step that **vindicated the "live smoke run" gate the project committed to in §6** — the gate that, until now, could be read as ceremony. It was not.

**The deploy.** `terraform apply` stood up **14 resources** — Aurora Serverless v2 (scale-to-zero) + the Data API, the S3 bronze bucket, the single Lambda, the EventBridge daily rule, SES, and the least-privilege IAM from [ADR-0014](adr/0014-operational-store-aurora-serverless-data-api.md). SES was verified; the schema was created on the live Aurora via **`alembic upgrade head` run over the Data API** (the same Data-API path the Lambda uses — not a psycopg2 side-channel). The infra came up exactly as the apply-then-destroy rehearsal in §22 promised it would.

**The pipeline ran end-to-end on real AWS.** One invocation returned **`statusCode 200`** with the full ledger: **fetched 10 → bronzed 10 → silvered 8 → gold 8 → scored 8 → email sent.** Real UAE Data-Engineering postings, scored against the live profile by `deepseek-v4-pro`, delivered to the inbox. **Two emails confirmed two validation gates *live*:**
- **VG5, both paths.** The first run at **threshold 60** produced **no matches** and sent a valid **"no matches today"** digest — the zero-path (§25) is a real output on real infra, not a swallowed no-send. An idempotent re-run at **threshold 20** produced a populated **7-job shortlist** — the matches-path.
- **VG4, live.** That re-run logged **`already: 8` skipped** — the `run_log` send-once guard saw the run-date was already processed and refused to re-send, exactly the per-run-date idempotency Step 7 wired in. Idempotency proven against the real database, not a mock.

Then **`terraform destroy`** tore down all 14 resources, verified gone, back to ~$0. We bought the proof and stopped paying for it — the same discipline as C-3 (§22), now applied to the whole system.

**The central lesson — the live run earned its keep, and the gate was never theater.** The end-to-end run caught **two bugs that *every* local test and *every* CI check structurally could not**, because both live on the Aurora **Data API** path that neither exercises — local tests run **psycopg2** against a container Postgres, CI runs a **postgres service**, and *neither speaks the Data API the deployed Lambda actually uses*:
1. **`migrations/env.py` — the configparser `%` interpolation bug (ERR-004).** Alembic ran the Data-API URL through `set_main_option`, whose configparser backend tried to interpolate the `%`-encoded ARNs (`%3A` in the cluster/secret ARNs) and choked. Invisible to psycopg2 (whose URL carries no ARNs). Fix: escape `%`→`%%` before handing the URL to configparser.
2. **`handlers/pipeline.py` `resolve_db_url` — the wrong connect-kwarg (ERR-005).** The URL was built with a `cluster_arn=` query param, but the `sqlalchemy-aurora-data-api` dialect maps query params straight to its `connect()` kwargs, where the parameter is named **`aurora_cluster_arn`**. Wrong name → `connect() got an unexpected keyword argument 'cluster_arn'` → this would have broken **every** deploy, not an edge case. Again invisible to psycopg2, which never touches that dialect.
Plus a sizing fix the live run surfaced: the Lambda timeout **300→900s**, because each posting costs ≈30s (one LLM call + one Data-API write) and the default 5-minute ceiling cut the run short. None of these three were reachable from the test pyramid as built — only real Aurora over real HTTPS exposed them. **This is the §6 "live smoke run" gate paying for itself:** a presence/liveness check is no gate (§15), and here the *behavioral* gate against the production-shaped target caught what structurally-mismatched tests (different driver, different URL) never could.

**The honest scale finding (a real P2 bottleneck for the roadmap).** The live run also surfaced a true limit, recorded plainly: **the single Lambda cannot run the full 18-query × 30-day backfill within the 15-minute Lambda max** — at ≈30s/posting across hundreds of historical postings, the one-time backfill blows the ceiling. The **daily incremental run (~10–30 new jobs) fits comfortably.** So v0 ships correct for its actual daily job; the backfill is a one-time job to **chunk**. This is not a defect — it is exactly the kind of *real* bottleneck P2 wants surfaced by use, and it **reinforces M3 (Step Functions / chunking)** as a justified future migration (the orchestration that survives §9's tool-minimalism precisely because real Lambda-duration complexity now earns it), with the backfill as its first concrete payload.

**So-what — v0 is shipped.** The evolutionary core is real: you wake to a scored shortlist of GCC Data-Engineering jobs, produced by a pipeline that fetched, landed, dissected, filtered, scored, and emailed — on live AWS, then torn down to ~$0. The build phase's through-line held to the last step: **intellectual honesty (ERR-004/005 logged with the real detection gap; the scale limit stated, not buried) and validate-on-real-infra-before-declaring-done** (the gate that looked like ceremony caught two deploy-breakers). The repo is the memory: **v0.1.0 is the irreducible core (§10), now built, deployed, validated, and tagged** — and from here the work is migrations, chosen one at a time by the bottleneck protocol (P2), with the first candidate (M3, surfaced by this very run) already on the board.

# Part 4 — The migrations, chosen by use (v0.2 → v0.6)

> Parts 1–3 designed, decided, and built the v0 core, then deployed it. Part 4 is the record of the **bottleneck protocol (P2) actually running** — five releases in a burst, each one chosen not from the pre-drawn M1–M8 roadmap but from **using the tool and feeling where it hurt**. The same rule holds: capture *what we chose, what we rejected, and why* — and this phase is unusually rich in **reversals**, because usage kept overruling the roadmap. Formal records live in [ADR-0021](adr/0021-m1-pipeline-hardening.md)–[ADR-0024](adr/0024-query-via-export.md), the [error log](ledgers/errors.md), and the per-release PRs; this is the narrative that ties them together.

## 28. The P2-driven capability burst (v0.2 → v0.6)
**Context.** §27 closed v0.1.0 with a promise, not a plan: the next work is *migrations, chosen one at a time by the bottleneck protocol*. Part 4 is that promise kept — five releases where **every single migration was picked by hands-on use, and the pre-drawn roadmap was treated as a hypothesis to be overruled, not a contract to be executed.** The through-line of the whole burst is one sentence: *the tool told us what to build next, repeatedly, and it was rarely what the roadmap had guessed.*

**M1 — the hands-on run that overruled the roadmap → pipeline hardening (v0.2.0).** The roadmap's pre-drawn hypothesis was **M1 = CV tailoring**. Before building it, we did the P2-mandated thing: **ran the tool live on the full 18-query GCC sweep to *feel* the bottleneck as a user, not guess it from the plan.** The run overruled the hypothesis flatly — the first real bottleneck wasn't tailoring a CV, it was **the pipeline completing at all.** Four distinct failures surfaced only under real load: **serial dissection timed out at silver ~17** (throughput, not correctness, was the wall); **a single DeepSeek `503` killed an entire run** (no failure isolation — one bad posting took down the batch); **the loose gold filter paid the pro-model to reject obvious junk** (spending the expensive LLM call to say "no"); and **AWS async auto-retry re-fetched the already-dead run** (a zombie-retry hazard, burning quota on a corpse). Fixed as three gate-trio units: **H-1** LLM retry+jitter and symmetric failure isolation (a provider blip skips *one* posting, never the run); **H-2** in-Lambda `ThreadPoolExecutor` concurrency (DB writes stay main-thread) + a deadline guard (runs return `partial`, never time out) + `maximum_retry_attempts=0` in Terraform + memory 512→1024; **H-3** a subset-title gold filter ("Data Architect" needs `data`+`architect`) made config-selectable behind the same `FilterStrategy` port (§23). **Re-validated on the exact ~132-posting backlog the pre-fix code died on:** `statusCode 200`, **~13× throughput** (~1.1→~14–15 dissections/min), 0 run-fatal crashes, a **21-job digest** delivered. *So-what:* this is the P2 protocol working *exactly as designed* — **usage, not the roadmap, chose the migration**, and the choice was defensible precisely because we had the live failure receipts. [ADR-0021](adr/0021-m1-pipeline-hardening.md); ERR-006/007.

**Make it his, not just Tarig's → user-customizable settings, then config-in-S3 (v0.3.0).** The next itch was ownership: could someone *else* use this? Auditing every parameter surfaced a pleasant finding — **the pipeline is already domain-agnostic**; an accountant could run it by editing config, nothing is hard-coded to Data Engineering. The real gap was that **settings were buried and frozen**: a **write-once trap** meant the profile row, seeded once at first run, never re-synced from config, so edits silently did nothing. The first fix was small and correct — **re-sync the profile row from config every run** + expose the **3 strictness knobs**. Then came the reversal that mattered. Tarig: *"editing a YAML and redeploying every time is not practical."* His first proposed fix (surface the settings as variables) only *half*-addressed it — it made the knobs findable but not *changeable without a rebuild*. Pushing past the symptom to the **root cause**: the config was **baked into the immutable Lambda zip**, so *any* change — variable or not — required a redeploy. The right fix was architectural, not cosmetic: **move config to S3, read it at runtime**, change settings via `push_config.py` with **no rebuild and no redeploy**. *Rejected:* **env-vars** (flat namespace, awkward for list-valued settings like queries/employment-types) and a **full settings UI** (that's the end-state, not the first step — deferred). We also adopted **patch-versioning** here (v0.3.1), whose first use immediately paid off: **v0.3.1 fixed a latent `employment_types` bug** — it was a free-text field that should have been an **enum**, and it wasn't actually **wired through to JSearch** at all. *So-what:* the reversal is the lesson — the user's proposed fix was accepted as a *signal* (the config is too hard to change) but not as the *solution*; chasing the root cause (immutable-zip config) produced a structurally better answer than the surface request would have. [ADR-0022](adr/0022-runtime-config-in-s3.md).

**"My progress should change my matches" → reassess / replay (v0.4.0).** The next use-driven ask: *a job I'm a stretch for today should become a strong fit once I learn the missing skill — and I want to see that without re-fetching anything.* This is a beautiful case of **usage re-deriving a designed-in capability from the other end.** Two things the architecture had *already* anticipated converged here: the medallion's **immutable-bronze → replay** property (§185/§23 — silver/gold/score are pure functions over bronze, so re-scoring costs **zero new API calls**) and the **graduation half of the old M4**, which the roadmap had filed abstractly and real use now made concrete. The schema had quietly prepared for it too — **`previous_score` and `save_score`'s auto-carry** existed before we had the feature that needed them. Built as a **`{"mode": "reassess"}`** invocation mode: re-run scoring over existing bronze against the updated profile. **Live-proven:** bumped Spark from *learning* → *expert*, re-ran, watched **15 jobs graduate** to higher scores — **bronze bytes unchanged.** *Rejected:* **re-fetch-and-rescore** (wasteful, and it *breaks the immutable-bronze value proposition* — the whole point is you don't pay the source again to re-derive), and **auto-reassess-every-run** (burns tokens re-scoring unchanged inputs; reassessment is an *event* — you learned something — not a schedule). *So-what:* the feature cost little because the medallion had been *built* for exactly this three sections and one release-phase before anyone asked for it — the immutable-bronze seam (§185) paying a dividend it was designed to pay. [ADR-0023](adr/0023-reassess-replay.md).

**"Let me query and filter my records" → export, not a UI (v0.5.0).** Tarig wanted to slice, search, and organize the accumulated postings. The tempting build is a custom filter/search UI; the **honest call was to refuse it.** The data is **SQL-shaped and tiny** — a custom UI would **reinvent, worse, what generic tools already do perfectly.** So v0.5.0 **exports to SQLite / CSV** and you open it in **Datasette or Excel** — full query/filter/sort power for near-zero build. A **hosted dashboard is the eventual end-state, not the first step** (same instinct as deferring the settings UI in v0.3.0). *So-what:* this is **P1 (absolute minimalism)** and the **defensibility rubric** at their sharpest — *"why this and not the simpler thing?"* answered by shipping the simpler thing on purpose, and naming the dashboard as a labeled future rather than building it because it would look good. [ADR-0024](adr/0024-query-via-export.md).

**"The email is poor, the links must be visible" → email UX (v0.6.0).** The digest that §27 was proud to have *delivered* turned out, in daily use, to be **hard to act on**: a dense table with the **Apply link buried**. §27's own closing note had already flagged the digest UX as the next candidate — usage confirmed it. Redesigned from a cramped table into **scannable per-job cards**, each with a **prominent Apply button** and the **location** surfaced, so the one action that matters (apply) is the one you can't miss. Built a **preview script** for a fast visual feedback loop — iterate on the rendered email without a full pipeline run. *So-what:* the smallest release of the burst, and still use-chosen — the tool's *own output*, met daily, named its own next fix.

**The cross-cutting reflections — three things the burst taught that outlive any one release.**
- **The roadmap is a living hypothesis, and the burst is the proof (P2).** Across five releases, **usage re-ranked the plan every single time**: M1 was *supposed* to be CV tailoring and the live run made it pipeline-hardening; reassess/replay and the query-export were re-derived from real asks, not executed from the M-list; the email fix came from reading a digest, not a plan. The pre-drawn **M1–M8 was hypothesis, never contract** — and this phase is the receipt that the bottleneck protocol (§P2) actually governs the roadmap rather than decorating it.
- **A recurring process scar — after changing any user-facing output *string*, grep *all* test files for the old string.** More than once, a wording change to an email line or a config-facing label turned CI red on **stale test assertions** still matching the *old* text. The change was correct; the tests were asserting the past. The discipline now: **change an output string → immediately grep every test file for the prior wording** before pushing. Cheap habit, repeated bite.
- **The stacked-PR mishap on M1 — merge stacked PRs strictly bottom-up.** M1's three gate-trio units (H-1/H-2/H-3) were built as **stacked PRs**, and the mistake was letting them **merge into their *base branches* rather than into `main`** — the stack folded into itself instead of landing on trunk. Caught and fixed, but the rule is now explicit: **stacked PRs merge strictly bottom-up** (base lands on `main` first, then the next re-targets `main`, and so on) — never let a child merge into a parent that hasn't itself reached trunk. A branching-hygiene lesson paid for in cleanup.

**So-what — the burst is the protocol, working.** Five releases, five migrations, **zero of them chosen by opening the roadmap and picking the next line.** Each was surfaced by *using the tool Tarig actually uses daily* — the timeout felt under real load, the frozen config hit while trying to change a setting, the stretch-job itch, the un-queryable pile of records, the un-actionable email. That is the dual purpose (§What-this-is) holding: **the tool's real friction is the portfolio's real roadmap.** The repo is still the memory — v0.6.0 is tagged, every reversal above is on the record with its *why*, and the next session resumes knowing not just *what* shipped but *that the choosing itself was done by use, as designed.*
