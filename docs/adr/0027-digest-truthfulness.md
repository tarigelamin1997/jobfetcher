# ADR-0027 — Digest truthfulness: new/still-open scoping, graduation visibility, render-time duplicate collapse

**Status:** Accepted · **shipped in v0.7.0's tag tree** (2026-07-08; live-validated 2026-07-09/10)
**Date:** 2026-07-08

## Context

The daily digest was **cumulative and unbounded**: `get_scored_shortlist` was deliberately global — its own SCOPE CAVEAT said so — so **every above-threshold job ever scored re-appeared in every digest**, the newest matches buried under repeats, a stale posting never expiring (Investigator-verified). Two more truths never reached the email: **reassess graduations** — ADR-0023's headline capability — existed only in the invoke-response JSON, so no email ever said "this job just crossed your bar"; and **within-JSearch duplicates** were each separately scored *and* separately carded — **measured 17.8% of scored postings** (32/180 in 14 fingerprint groups; worst case: "Data Platform Engineer @ Blackstone EIT" scored **5×** with scores 35–88), every dup paying the pro-model to be judged again and occupying its own card. A digest that re-presents old news, hides graduations, and cards the same role five times is a digest that lies.

## Decision

Make the digest tell the truth **render-side** — **no migration, no new table** (everything rides existing columns: `score.scored_at`/`previous_score`, `posting.fingerprint`, `run_log.digest_sent_at`). Commits `7364cc7` + `0eca0f7`; **shipped inside the `v0.7.0` tag tree** (merged before the tag was cut; credited to v0.7.0 — see the CHANGELOG release-numbering note).

- **The new/still-open rule (the Examiner-corrected version — see Consequences):** an item is **NEW** iff `since is None` (the first-ever digest — no `run_log` rows, everything is new) OR `scored_at is None` (defensive: an unknown judgment time never silently demotes a match) OR (`scored_at > since` AND (`previous_score is None` OR `previous_score < threshold <= score`)) — i.e. a **fresh judgment that is actually news**: a first scoring, or a graduation. Everything else above threshold is **STILL OPEN** — the daily repeat (scored before the last digest), the fresh-but-non-graduated re-score (being re-judged is not news), and a graduation an earlier digest already announced.
- **`since` = the new `Repository.get_last_digest_sent_at(user_id)`** — `MAX(run_log.digest_sent_at)`, NULL-safe (`None` = first-ever) — resolved **once** by `notify()` and threaded to both the shortlist read and the renderer.
- **`get_scored_shortlist` gains `since` + `max_age_days`** (both declared on the `Repository` Protocol, as is `get_last_digest_sent_at`); its SELECT adds `previous_score`, `fingerprint`, `scored_at`, and the effective fetched-at; `ShortlistItem` carries the four new fields. The split and grouping stay **pure render-side functions** (`split_new_and_still_open` / `collapse_duplicates` in `core/notifier.py`) — no SQL-side splitting.
- **New REQUIRED `SearchSpec` knob `digest_max_age_days`** (`ge=0, le=365`; `0` = no cutoff; sample recommends **90**): a still-open match older than N days drops out of the digest entirely. Age = **`COALESCE(posting.fetched_at, bronze_posting.fetched_at)`** (the ADR-0025 reassess pattern — bronze is the effective live source); **unknown-age rows INCLUDED**, never silently dropped; the cutoff is a **bound parameter**, never interpolated SQL. **Documented semantics change:** the `below` count is now scoped to the same age window — an aged-out job vanishes from the digest entirely, footer included.
- **Renderer:** **"New since last digest"** full cards lead; a graduation carries a green **text** badge **`↑ old→new`** (inline styles, no images — email-client-safe). Same-fingerprint groups collapse to **one card** (representative = the highest-scoring member; footnote `seen n× — scores lo–hi`; None/empty fingerprints are never merged); grouping runs over the **whole** surfaced set and each group renders in exactly **one** section (NEW iff any member is new) — a straddling group never renders twice. **"Still open"** = a count line + the top-5 compact one-liners + an "…and n more — see your export" overflow ([querying.md](../querying.md)). A zero-new day honestly says **"no new matches since {date}"** and **still sends** (the VG5 spirit: never skipped, never blank); the subject counts **collapsed groups**, not raw rows.
- **`notify()` stays the single threshold authority** and resolves `since` once; `handlers/pipeline.py` threads `spec.digest_max_age_days`; `scripts/preview_digest.py` now previews all three new states (a graduation badge, a seen-3× dup group, the still-open section).
- **Cluster-attach dedup is explicitly deferred to M2** (Tarig's scope call): render-collapse fixes the user-facing clutter *now*; attaching same-fingerprint postings to one cluster — which would stop the duplicate **scoring spend** at the source — belongs to the multi-source clustering redesign ([ADR-0005](0005-dedup-cluster-and-surface.md)'s machinery), not to a render fix.

## Alternatives considered

- **Keep the global digest (the status quo).** Rejected: it defeats daily usability — the digest's one job is "what's worth my time *today*", and a cumulative, unbounded list answers a different (and already-answered, via the export) question.
- **`previous_score`-based newness (the original spec).** Rejected — as a **corrected defect**, not a road untaken: in daily-only operation a posting is scored exactly once, so `previous_score` stays NULL forever and the rule re-presents **every old job as "New" every day** — the exact disease this unit exists to cure. The signal is the judgment's freshness (`scored_at` vs the last digest), not the score's ancestry.
- **Cluster-attach dedup now.** Rejected: crosses into M2's cluster redesign (real clustering, `match_status`, the Suspected-Duplicates surface) for a render-time problem; deferred to M2 by explicit scope call.
- **`digest_max_age_days` optional-with-default.** Rejected: violates the `SearchSpec` every-field-explicit / fail-loud contract (the same rule that made `hard_floor`/`near_miss_band` and `reassess_max_age_days` required).
- **A "seen"/digest-membership state table.** Rejected: a mutable read-model duplicating what `score.scored_at` vs `run_log.digest_sent_at` already encodes — anti-P1, and a fresh instance of the in-place-state pattern migrations 0004/0005 just removed.

## Consequences

- **The daily email is news again** — new matches lead, graduations are finally *visible* in the email (closing the "the digest of these rides the email-UX unit" promissory note from [ADR-0023](0023-reassess-replay.md)), a duplicated role costs one card instead of five, and stale postings age out instead of accumulating forever.
- **A process win worth recording honestly:** this was the **first auto-pilot unit** under the severity-gated policy adopted 2026-07-07 (Claude merges on a clean Examiner pass + green CI, no per-unit human gate). The original spec carried the `previous_score`-only newness defect above; the **independent fresh-context Examiner caught it** (a *contested* finding, adjudicated fix-now), the rule was corrected (`0eca0f7`), and the re-verify came back **CLEAN PASS**. The severity gate did exactly what it was adopted to do.
- **Known cosmetic corners (recorded, not hidden):** (1) a straddling dup group made NEW by a fresh twin uses its highest-scoring member as representative — so a *stale* graduation badge can re-render on that card (factually true, narrow); (2) `scored_at == since` exactly classifies still-open (the correct direction; not pinned by a test); (3) the plaintext dup footnote lists the collapsed raw posting ids while the HTML omits them (a deliberate plaintext-debuggability asymmetry).
- **⚠️ Deploy sequencing (the ADR-0025 F1-class again):** `digest_max_age_days` is REQUIRED and the runtime config lives in S3 ([ADR-0022](0022-runtime-config-in-s3.md)) — the deployed S3 config **must be re-pushed** (`scripts/push_config.py`) before the next invoke after this deploys, or **every run fails loudly** (`SearchSpec` ValidationError). Same order as v0.7.0's `reassess_max_age_days`: **update the local config YML → `push_config.py` → deploy/invoke** (registered in the [procedure registry](../ledgers/procedure-registry.md)).
- Verified: **311 unit green** (+28 net over Run 2's 283); **39 integration passed + 5 live-key skips** (local Postgres window, 2026-07-08); full suite **350 passed, 95.02% coverage** (85% floor); `ruff` clean; independent fresh-context adversarial Examiner: initial pass **contested F1 → fixed → re-verified CLEAN PASS**.
