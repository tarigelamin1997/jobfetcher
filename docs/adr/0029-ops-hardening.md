# ADR-0029 — Ops hardening: S3 remote state · dead-man + errors alarms · the `{"mode":"smoke"}` deploy gate

**Status:** Accepted · **shipped v0.9.0** (2026-07-10)
**Date:** 2026-07-08

## Context

Three operational risks had accumulated under eight releases of capability work, none of them visible in a test suite:

1. **The Terraform state lived only on this laptop.** A disk loss or reimage meant hand-cleaning a live **14-resource stack** — orphaned Aurora/Lambda/S3 no state tracks — and the state file itself holds the **account id, emails, and every ARN**: sensitive material with no backup and no locking.
2. **Zero alarms.** A silently-dead cron (schedule disabled, rule deleted, permission broken) had exactly one signal: **a missing email** — indistinguishable from an honest "no matches today." The failure mode a daily-digest tool can't self-report is silence.
3. **No post-deploy gate for the ERR-004/005 class.** Data-API-only failures are invisible to local psycopg2 tests *and* CI's postgres service ([ERR-004/005](../ledgers/errors.md)) — so every release accumulated live "watch items" verified by hand, and a schema-vs-code mismatch surfaced at the next morning's failed run at the earliest.

## Decision

Fix all three as one **ops-hardening** unit — **Squad Run 5, the final run of the squad program**. Commits `7b724ce` (terraform) + `09335b2` (handler + tests) + `2c48b1b` (runbook) + `a359431` (Examiner fixes); **shipped as v0.9.0** (2026-07-10) after Tarig's PR review + the one-time state migration. **CRUCIAL tier — PR-reviewed by Tarig.**

- **S3 remote state** ([providers.tf](../../terraform/providers.tf)): a **`backend "s3"`** block — bucket **`jobfetcher-tfstate-198592435375`, deliberately UNMANAGED** (created once outside the stack, never a resource in this config) because the state must **survive `terraform destroy`** — the end-of-day teardown cadence makes a stack-managed state bucket a chicken-and-egg: destroying the stack would delete the record of its own teardown (the Investigator's draft had exactly that; the orchestrator caught it). Locking is **`use_lockfile = true`** — TF ≥ 1.10 **native S3 locking**, no DynamoDB lock table — so `required_version` bumps **1.6 → 1.10** accordingly. The **ONE-TIME local→S3 migration is human-present** ([deploy runbook §3](../runbooks/deploy.md)): **backup first → `terraform init -migrate-state` → a ZERO-drift `terraform plan` as the stop condition → local-file removal last.**
- **Two CloudWatch alarms → one SNS topic → email** ([alarms.tf](../../terraform/alarms.tf); endpoint `var.recipient_email`; the subscription delivers nothing until the **confirmation click** — [runbook §4](../runbooks/deploy.md); new **`alarms_topic_arn`** output for the verify step, Examiner S1):
  - **Dead-man** — `AWS/Events Invocations` on the daily rule, dimension **`RuleName`** (the Surgeon corrected the spec's wrong `Rule` key: with missing-data-as-breaching, a wrong dimension key would have **permanently false-fired** — the metric it watched would never exist); **period 86400 × 1 evaluation — the ONLY legal full-day shape** (CloudWatch caps period × evaluations at one day); `Sum < 1`; **missing data = breaching** (Invocations is a sparse metric — missing IS the dead signal). **Honest lag semantics:** 1-day UTC windows are judged at close, so detection lands up to **~2 days after the last good run** — acceptable for a daily digest; tighter needs more moving parts than the problem earns (P1).
  - **Errors** — `AWS/Lambda Errors` (crashes/timeouts), 3600 × 1, `Sum >= 1`, `notBreaching` on missing (no invocations ≠ a crash — silence is the dead-man's job). **Honest gap, documented:** the handler *returns* `statusCode: 500` on a stage failure — a *successful* invocation to Lambda, invisible to this alarm (a custom metric is the documented future refinement).
  - Both alarms notify **recovery** too (`ok_actions`).
- **`{"mode":"smoke"}` — the post-deploy gate** ([handlers/pipeline.py](../../src/jobfetcher/handlers/pipeline.py)): at the **very top of the handler `try`**, before any config read or adapter construction — env + a Data-API connect + `SELECT version_num FROM alembic_version` compared to the head this code expects (**`$ALEMBIC_HEAD`** env pin, set by terraform per migration; hardcoded fallback `0006_subscores`; a **staleness unit test** pins the constant to the *real* migrations head via alembic's `ScriptDirectory` — a new migration without the bump fails unit CI, not the next deploy). **200** match / **400** wrong-head (names both sides) / **500** unreachable-or-never-migrated (the `error` field disambiguates schema-vs-infra — Examiner N1's runbook clause). Touches **no** config/LLM/S3/email and **writes nothing** — the integration tests prove **zero rows across all 8 tables** with exploding fakes on all six adapter symbols; < 2 s warm (a scale-to-0 Aurora resume can add ~30 s). The **ERR-008-compliant one-liner** lives in [runbook §2](../runbooks/deploy.md) as release-sequence step 6: **200 or you stop.**
- **`scripts/invoke.py` (the ERR-008 wrapper): explicitly decided AGAINST for this run** — the invocation pattern stays **procedure-only** (the [registry row](../ledgers/procedure-registry.md) updated by the Surgeon; the runbook one-liners encode `AWS_MAX_ATTEMPTS=1` + `--cli-read-timeout`). A wrapper waits until real ops friction proves the procedure insufficient.

## Alternatives considered

- **A DynamoDB lock table** (the classic remote-state companion). Rejected: superseded by TF ≥ 1.10's **native S3 locking** (`use_lockfile`) — an extra managed table + IAM surface for a problem the backend now solves itself.
- **A Terraform-MANAGED state bucket** (including the reuse-the-data-bucket-under-a-prefix variant). Rejected: **chicken-and-egg with the destroy-to-zero cadence** — the bucket holding the state cannot be a resource *inside* that state; `terraform destroy` would delete the record of its own teardown. (The Investigator's draft carried this defect; caught before build.)
- **A CloudWatch dashboard now.** Rejected: nothing to stare at daily — the tool's failure model is *silence* and *crash*, and two alarms are the P1-minimal *observable*; a dashboard is M7's conversation.
- **A custom metric for returned `statusCode`-500s now.** Rejected: the Errors alarm covers crashes/timeouts; the returned-500 gap is documented as the known future refinement — a metric filter now is machinery ahead of a real miss.
- **The `scripts/invoke.py` wrapper (ERR-008).** Rejected for this run: the procedure (runbook one-liners + the registry row) suffices until ops friction proves otherwise.
- **Smoke checks beyond schema-version** (row counts, config validation, LLM/SES reachability). Rejected: the gate targets **exactly** the deploy-only failure class ERR-004/005 proved (Data-API reachability + schema-vs-code head) — nothing more; every extra check adds side-effect surface to a probe whose contract is *zero side effects*.

## Consequences

- **The ERR-004/005 watch-item cycle becomes a mechanical gate.** Every release since v0.1.0 verified Data-API-only behavior by hand-watching the first live run; now a 5-second `{"mode":"smoke"}` invoke pins *deployed code ↔ migrated schema* at step 6 of the release sequence — a wrong head is a **400 at deploy time**, not a failed morning run.
- **The state's blast radius is now explicit and contained** — the tfstate holds the account id, emails, and every ARN; it lives in one private, encrypted, **versioned** bucket (versioning is the recovery path), never the repo. Laptop loss stops being a stack-orphaning event.
- **Known benign quirk (Examiner N2, recorded):** a `destroy` → `apply` cycle that straddles the 06:00 UTC cron can **false-fire the dead-man on day one** (the empty pre-apply window is judged at close); it **self-clears** at the next window and the recovery `ok_action` announces it — accepted rather than special-cased.
- **SNS pending-subscription expiry (Examiner N4, recorded):** an email subscription left `PendingConfirmation` delivers nothing — the alarms fire into the void — and an unclicked confirmation **expires** (~3 days). The runbook's verify step (a real ARN, not `PendingConfirmation`) plus the **re-check after every `destroy`→`apply`** (recreating the subscription re-requires the click) make the confirmation part of the procedure, not a memory.
- **Live-smoke watch item (this release's own):** the **first live `{"mode":"smoke"}` invoke itself** — the smoke SELECT's **`scalar_one()` over the Aurora Data API is a codebase first** (a new dialect-result surface: exactly the ERR-004/005 class the gate exists for). Registered in the [procedure registry](../ledgers/procedure-registry.md).
- Verified: **349 unit green**; **45 integration passed + 5 live-key skips** (local Postgres window, 2026-07-08); full suite **394 passed, 95.21% coverage** (85% floor); `ruff` clean; `terraform validate` clean; independent fresh-context adversarial Examiner: **CLEAN PASS, zero blocking** (its S1 `alarms_topic_arn` output + N1 400/500-boundary runbook clause applied in `a359431`; its RuleName-dimension, 86400×1-shape, and unmanaged-bucket verifications all confirmed).
