# ADR-0028 — Scorer subscores + shadow code-total: the 7-factor formula exists at last

**Status:** Accepted · **shipped v0.8.0** (2026-07-10)
**Date:** 2026-07-08

## Context

DeepSeek `v4-pro` is **non-deterministic even at temp 0** (the VG3 best-effort finding, accepted since v0.1) — and the lineage log ADR-0025 built made the magnitude *measurable* for the first time. **Measured (Investigator-verified, live lineage data):** same-input re-scores drift by deltas up to **27+**, and one same-fingerprint job scored **35–88** across runs. The `near_miss_band` is **10** — so a boundary categorization (`strong_fit` vs `near_miss` vs `stretch`) was **partly random**: the band is narrower than the noise.

The deeper finding: the **7-factor weights existed NOWHERE**. The architecture doc said *"weights tunable; refined during the scoring migration"* (i.e. TBD), no code carried a weight table, and the prompt told the LLM *"the score is your holistic judgment"* — the model had been **freelancing the weighting since v0.1**. A "7-factor framework" whose factors have no weights isn't a formula; it's a vibe with a rubric.

## Decision

Decompose the score into **7 per-factor subscores**, write the weight formula down **in code for the first time**, and run the code-side total in **SHADOW mode** — observed, logged, persisted, but **never the product number** until M7 says so. Ships as migration **`0006_subscores`** (chains to `0005_application_event`); commits `168ab39` + `33a8ef4` + `cda34b9`; **shipped as v0.8.0** (2026-07-10). **CRUCIAL tier under the severity-gated policy — PR-reviewed by Tarig** (unlike Run 3's auto-pilot merge).

- **`ScoreResult` gains 7 OPTIONAL bounded (0–100) subscore fields** — `core_skill_match`, `tool_tech_alignment`, `achievement_relevance`, `seniority_scope`, `ats_keyword_coverage`, `domain_sector_fit`, `realistic_fit`.
- **`FACTOR_WEIGHTS` ([core/scorer.py](../../src/jobfetcher/core/scorer.py)) — the formula's first documented existence.** Chosen by Tarig 2026-07-08; provenance: the original draft-PDF's ATS scoring framework, adapted. **core_skill_match 0.30 · tool_tech_alignment 0.20 · achievement_relevance 0.15 · seniority_scope 0.15 · ats_keyword_coverage 0.10 · domain_sector_fit 0.05 · realistic_fit 0.05.** A **module-load `assert sum == 1.0`** makes a mis-edited table fail the import, not skew every total (verified exact in IEEE-754 *as ordered* — see Consequences for the Examiner's order-fragility note).
- **Prompt:** the scoring prompt now requests the 7 subscores alongside the holistic score. **`score` is demanded FIRST in the output order** — an autoregressive anti-anchoring measure: the holistic number is emitted before any subscore token exists to anchor on — plus an explicit *"not a mechanical average"* instruction. **Zero pre-existing prompt instructions changed** (the Examiner AST-diffed the runtime strings to prove it).
- **SHADOW mode (Tarig's explicit choice):** `score` — the LLM's holistic total — **remains the product number everywhere**: banding, threshold, digest all untouched (Examiner grep-proved no leak; the hardened invariant test discriminates **every** consumer path — persisted score, `fit_category`, and surfaced-count each independently catch a `code_total` leak). **`compute_code_total`** = the pure weighted sum with **banker's rounding** (Python `round`, half-to-even — pinned exact: 73.5→74, 72.5→72), logged as drift telemetry (`INFO`; `WARNING` when |delta| > 20). **Cut-over to `code_total` is an M7 decision** after ~a week of reassess data — never a side effect of this unit.
- **Migration 0006:** additive nullable **`subscores` JSONB** on `score` **and** `score_event` (no backfill — pre-0006 rows stay NULL). The persisted blob = **`{7 factors, code_total, llm_total}` — or NULL when the LLM omitted them, never a partial dict**. A re-score **REPLACES** the current-view blob, **even with NULL** — a stale breakdown must not misdescribe a new judgment; history is safe on the self-contained `score_event` rows.
- **Failure asymmetry (deliberate):** **OMITTED** subscores fail **open** — the blob stays NULL, a `"subscores_missing"` telemetry line fires, the run continues (a missing breakdown never costs a scoring). **MALFORMED / out-of-range** subscores fail **closed** — the Pydantic bounds reject them into the existing retry→skip path (a lying breakdown never gets persisted).
- **The reassess report becomes the variance instrument:** `reassess()` gains **`delta_buckets`** (`"0-5"` / `"6-10"` / `"11-20"` / `"21+"`) + **`max_delta`** + **`mean_delta`** (1-decimal) over successful re-scores — the same-input drift that motivated this unit is now measured by every replay, for free.
- **Test-infra fix that rode along (`cda34b9`):** `migrations/env.py` called bare `fileConfig(...)` — Python's default `disable_existing_loggers=True` **silenced every imported `jobfetcher.*` logger** after any programmatic alembic use, surfacing as an order-dependent caplog failure in the full suite. Fixed with `disable_existing_loggers=False` + order-independence hardening in the test — protects **all** future caplog tests.

## Alternatives considered

- **Replace the product number with `code_total` immediately.** Rejected: changes the core number of the whole tool with **zero observation data** on how the code total behaves on live jobs. Shadow first, cut over on evidence — the same observe-then-migrate discipline as every other release.
- **Telemetry-only (log the subscores, persist nothing).** Rejected: loses the factor-level trail M7's calibration needs — per-factor drift, code-vs-LLM deltas per cohort. Unpersisted telemetry is exactly the unreplayable-loss shape ADR-0025/0026 exist to prevent.
- **Keep-stale-blob / COALESCE on re-score.** Rejected: a current-view breakdown desynced from its own score — the blob would describe a *previous* judgment while sitting next to the new number. The current view replaces; the events keep history.
- **N-sample averaging (score each job 3×, average).** Rejected: 3× cost + latency for a noise reduction that's statistically modest at N=3 — anti-P1 (and rejected once already, back in v0.1's VG3 decision).
- **Switching models for determinism.** Rejected: deviates from [ADR-0017](0017-llm-transport-openai-compatible-deepseek.md) for an unproven gain — MoE/FP non-determinism at temp 0 is not provider-unique, and the shadow instrument works on any backend.
- **Rounding: half-up instead of banker's.** Considered; Python's default `round` (half-to-even) kept — deterministic, tie-unbiased over an accumulating calibration corpus, and zero custom code; the exact behavior is pinned by test (73.5→74, 72.5→72) so nobody re-litigates it by accident.
- **Integer-percent weights (30/20/15/15/10/5/5, sum == 100).** Named as the **cleaner future alternative** — the Examiner (N3) showed the float sum assertion is order-fragile (952 of the 5040 possible orderings would fail `== 1.0` in IEEE-754). Accepted as-is: the current order is verified exact, and the failure mode is **loud at import**, never a wrong total.

## Consequences

- **The scoring formula is finally inspectable, testable, and versioned** — "why did this job score 72?" now has a factor-level answer, and M7 calibration gains its data source: a code-vs-LLM delta corpus with per-factor resolution, accumulating from the first post-deploy run.
- **The M7 cut-over criterion (stated now, decided then):** if the code-total spread on same-input re-scores stays **≤ the near-miss band**, tighten VG3 from best-effort toward a real tolerance and consider cutting the product number over to `code_total`. The reassess `delta_buckets` are the evidence feed.
- **Anchoring assessment (Examiner N5, recorded honestly):** score-first ordering + "not a mechanical average" mitigate *within-output* anchoring, but the residual risk is **prompt-change drift of the holistic score itself** — adding subscore demands may shift the distribution the product number is drawn from. This is **measurable, not speculative**: the first post-deploy reassess re-scores the same inputs under the new prompt, so its delta distribution **doubles as the old-vs-new-prompt instrument** (registered as a live-smoke watch item).
- **⚠️ Deploy sequencing (Examiner S2 — STRICTER than the previous class):** the new `save_score` writes the `subscores` column **UNCONDITIONALLY** — deployed against a DB not migrated to 0006, **every scoring fails** (run-fatal for the stage — not the loud-and-harmless `UndefinedTable` of the 0005 script class). The order is: **`alembic upgrade head` (0006) BEFORE deploying this code.** Read-side is unaffected — the export selects explicit columns. (Registered in the [procedure registry](../ledgers/procedure-registry.md).)
