# Ledger В· Interface Contracts (Produces в†’ Consumes)

> The single source of truth for what each stage **emits** and who **consumes** it. When a release closes, verify its *Consumes* against already-shipped *Produces*, then append its *Produces* row. This prevents cross-stage drift in a multi-stage pipeline вЂ” and doubles as lineage documentation.

**Status:** **v0.12.0 SHIPPED (tag `v0.12.0`, 2026-07-17) вЂ” every contract below is вњ… built + shipped + live-validated on the deployed stack.** The pipeline now runs **fully unattended** (the daily 06:00 UTC EventBridge cron drives the whole run end-to-end with nobody watching; first solo flight 2026-07-10). The **v0 contracts** (all live-validated at v0.1.0 over the Data API): the **`SearchSpec`** search input, the **`DissectedPosting`** silver dissection contract (C-1), the **`Profile`** contract + the **gold-filter** Produces (Unit A), the **`score`** contract (Unit B / the Scorer), the **`notify`** Produces (Step 6 вЂ” the daily SES digest, with ADR-0015's previously-undefined **`Notifier`** port now defined), the **handler / orchestration** Produces (Step 7 вЂ” one Lambda wiring the stages + the `run_log` send-once contract), and the **`schema`** (C-2). **Nine P2-driven releases** append their Produces in the second table below, through **v0.12.0** (full S3 audit persistence вЂ” every stage's structured results also land in S3 as batched JSONL вЂ” + a local Streamlit control panel). The single-Lambda orchestration met the daily incremental run but **can't** drive the full 18-query Г— 30-day backfill inside the 15-min cap в†’ reinforces **M3** (Step Functions).
