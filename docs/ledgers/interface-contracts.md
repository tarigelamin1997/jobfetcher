# Ledger · Interface Contracts (Produces → Consumes)

> The single source of truth for what each stage **emits** and who **consumes** it. When a release closes, verify its *Consumes* against already-shipped *Produces*, then append its *Produces* row. This prevents cross-stage drift in a multi-stage pipeline — and doubles as lineage documentation.

**Status:** **v0.1.0 SHIPPED (tag `v0.1.0`, 2026-06-29) — every v0 contract below is ✅ built + shipped + live-validated** (deployed to AWS, ran end-to-end over the Data API, destroyed to ~$0). The contracts: the **`SearchSpec`** search input, the **`DissectedPosting`** silver dissection contract (C-1), the **`Profile`** contract + the **gold-filter** Produces (Unit A), the **`score`** contract (Unit B / the Scorer), the **`notify`** Produces (Step 6 — the daily SES digest, with ADR-0015's previously-undefined **`Notifier`** port now defined), the **handler / orchestration** Produces (Step 7 — one Lambda wiring the stages + the `run_log` send-once contract), and the **`schema`** (C-2). The single-Lambda orchestration met the daily incremental run but **can't** drive the full 18-query × 30-day backfill inside the 15-min cap → reinforces **M3** (Step Functions). Future-migration contracts append below as each ships (⬜).
